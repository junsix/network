08 · Trust

TLS — 도청 가능한 인터넷에서 안전한 채널 만들기

TLS는 세 가지를 동시에 풀어냅니다: 기밀성(내용을 못 봄), 무결성(중간에 못 고침), 인증(상대가 진짜인지 확인). 이 세 가지를 비대칭 키 교환 + 대칭키 암호화 + 인증서 체인으로 엮어 만들어 냅니다.

아이디어

핵심 한 줄

공개키로 키 교환을 안전하게 협상한 뒤, 실제 데이터는 빠른 대칭키로 암호화한다.

공개키 암호는 인증·키 교환에는 강하지만 느립니다. 대칭키는 빠르지만 둘이 같은 키를 가지고 있어야 합니다. TLS 핸드셰이크는 "느린 공개키로 빠른 대칭키를 안전하게 합의"하는 절차입니다.

Server
  1. ClientHello: 지원 가능한 암호 스위트 목록 + 클라이언트 키 공유(KeyShare) + 무작위 값(ClientRandom).
  2. ServerHello: 선택한 암호 스위트 + 서버 KeyShare + ServerRandom. 여기서 양쪽은 ECDH(타원곡선 디피-헬만)로 공유 비밀을 계산 가능. 이후 메시지는 이미 암호화된다.
  3. Certificate (암호화됨): 서버 인증서 체인. 클라이언트는 신뢰 루트까지 거슬러 올라가 검증.
  4. CertificateVerify: 서버가 자기 개인키로 서명 → "이 인증서는 정말 내 것이다" 증명.
  5. Finished: 양쪽 모두 핸드셰이크 무결성 확인.
  6. 이후 모든 애플리케이션 데이터는 합의된 세션 키로 AEAD(AES-GCM 등) 암호화.
PFS

왜 디피-헬만(DH)을 쓰나 — 순방향 비밀성

과거 RSA 키 교환에서는 서버 개인키만 알면 과거 트래픽까지 복호화할 수 있었습니다. 오늘날에는 매 세션마다 임시 ECDHE 키를 만들어, 세션 종료 후엔 그 키가 사라집니다. 나중에 서버 개인키가 유출돼도 과거 트래픽은 복호화 불가 — 이것이 PFS입니다.

신뢰

인증서 체인과 신뢰 모델

브라우저가 신뢰하는 Root CA 목록은 OS와 브라우저에 미리 들어 있습니다. 그 사슬이 어디 한 곳이라도 깨지면 (만료, 폐기, 도메인 불일치) 빨간 경고가 뜨는 것이죠.

한계

강점과 한계

TLS가 보장하는 것

  • 도청 방지: 중간에 패킷을 봐도 내용은 암호문
  • 위변조 탐지: 한 바이트만 바꿔도 AEAD가 깨짐
  • 신원 확인: 인증서를 통해 "이 도메인의 진짜 서버"임을 검증
  • 순방향 비밀성: 과거 통신을 미래의 키 유출로부터 보호

TLS가 보장하지 못하는 것

  • 경로 자체의 차단/리다이렉션 (그건 ARP·DNS·라우팅 보안의 몫)
  • 서버가 신뢰할 만한지 (인증서는 "맞다"는 것만 보장)
  • 클라이언트 자체의 무결성 (악성 코드가 깔리면 의미 없음)
  • CA가 잘못 발급한 경우 (Certificate Transparency, HSTS 등으로 보완)
