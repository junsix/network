HTTP — 웹의 언어, 세 번의 큰 진화

HTTP는 "요청 → 응답" 모델의 단순한 텍스트 프로토콜로 시작해서, 성능과 보안의 압력을 받으며 HTTP/2(바이너리 멀티플렉싱), HTTP/3(QUIC 위)로 진화해 왔습니다. 각 버전이 "어떤 문제를 풀려고 했는지"를 따라가 보면 설계 의도가 보입니다.

기본 형태

요청과 응답의 모양

GET /index.html HTTP/1.1
Host: example.com
User-Agent: curl/8.4.0
Accept: */*

HTTP/1.1 200 OK
Content-Type: text/html; charset=utf-8
Content-Length: 1256
Cache-Control: max-age=3600

<!doctype html> ...

첫 줄(요청 라인 또는 상태 라인), 헤더, 빈 줄, 본문. 텍스트라 어떤 도구로도 들여다보기 쉽습니다. 이 단순함이 HTTP의 성공 비결이지만, 이후 성능 문제의 근원이기도 합니다.

HTTPS

HTTP 위에 TLS

HTTPS는 별도의 프로토콜이 아닙니다. HTTP를 TLS 위에서 전송하는 것뿐입니다. 그래서 도청·위변조·신원 사칭에서 자유로워지지만, 핸드셰이크 비용과 인증서 운영 부담이 생깁니다.

자세한 TLS 핸드셰이크는 TLS 페이지에서 다룹니다.

강점과 약점

HTTP가 잘 어울리는 곳

  • 요청/응답 모델로 충분한 모든 것 (REST API, 정적·동적 웹)
  • 방화벽·CDN·캐시 인프라가 이미 풍부함
  • 텍스트 친화로 디버깅·관측이 쉬움

HTTP가 어색한 곳

  • 서버가 먼저 푸시해야 하는 실시간 (해결: WebSocket, SSE, gRPC streaming)
  • 지속적·짧은 메시지가 많은 게임 (HTTP의 오버헤드)
  • 방화벽/프록시가 HTTP만 통과시키는 환경에서 다른 프로토콜을 쓰고 싶을 때 (HTTP CONNECT, WebSocket, 또는 HTTPS 터널 사용)
