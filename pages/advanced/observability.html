관측 — 보이지 않으면 막을 수 없다

장애가 났을 때 "어디가 느린지", "누가 누구에게 무엇을 보냈는지"를 모르면 어떤 방어도 추측에 불과합니다. 관측 도구는 계층마다 다릅니다 — 비트를 직접 들여다보는 것부터, 의미 있는 트레이스를 따라가는 것까지. 어디서 어떤 도구를 쓰는지 알면 디버깅이 훨씬 빨라집니다.

스택

관측의 계층 지도

위로 갈수록 의미 풍부, 아래로 갈수록 사실 풍부. 가설은 위에서, 검증은 아래에서가 좋은 패턴입니다.

패킷

tcpdump — 가장 정확한 진실

커널이 NIC에서 패킷을 직접 캡처합니다. BPF 필터로 관심 트래픽만 추립니다. 

# 80포트로 가는/오는 트래픽, 호스트 1.2.3.4에 한정, 패킷을 파일로
sudo tcpdump -i any -nn -w /tmp/c.pcap 'host 1.2.3.4 and port 80'

# 그 자리에서 헤더 + 페이로드 일부 보기
sudo tcpdump -i any -nnvvXSs0 'tcp port 443 and host api.example.com'

# DNS 질의/응답만
sudo tcpdump -i any -nn 'udp port 53'

# TLS handshake만 (Client Hello 시작 바이트 패턴)
sudo tcpdump -i any 'tcp port 443 and (tcp[((tcp[12:1] & 0xf0) >> 2):1] = 0x16)'
프로덕션에서는 패킷이 너무 많아 디스크가 폭증할 수 있습니다. -c 1000(N개만), -G 60 -W 10(60초씩 10개 로테이션) 같은 옵션으로 제한하세요.
Wireshark

패킷을 의미 있는 흐름으로

Wireshark는 동일한 캡처를 GUI로 분석합니다. 가장 강력한 건 display filter와 follow stream:

http.response.code >= 500
5xx 응답만 모아 보기
tcp.analysis.retransmission
재전송 발생 패킷 — 손실·혼잡 분석의 첫걸음
tls.handshake.type == 1
ClientHello만. SNI·암호 스위트 확인
dns.qry.name contains "example"
도메인 부분 매칭
Follow → TCP Stream
한 4-tuple의 송수신 바이트를 한 창에 재구성. HTTP 요청·응답 검사에 유용
Statistics → I/O Graph
시간별 처리량, 윈도우, RTT 시각화
흐름 통계

NetFlow · sFlow · IPFIX

패킷을 모두 캡처하면 너무 많습니다. 흐름(flow) 단위로 요약합니다 — 같은 5-tuple(src/dst IP, src/dst port, proto)의 패킷 묶음을 "X 바이트, Y 패킷, Z초 지속"으로 집계. 라우터·스위치가 이 요약을 컬렉터로 보냅니다.

eBPF

커널 안에서 측정하기

eBPF는 검증된 작은 프로그램을 커널 핫패스에 안전하게 끼워 넣게 해주는 기술입니다. 성능 비용 거의 없이 — tcpdump보다 훨씬 — 커널이 보는 모든 이벤트를 관측할 수 있습니다. 

# BCC/bpftrace 유틸들
sudo tcpconnect-bpfcc          # 신규 TCP 연결 실시간 로그
sudo tcpretrans-bpfcc          # 재전송 발생 위치·시간
sudo tcpaccept-bpfcc           # listen 측 수락 이벤트
sudo tcplife-bpfcc             # 연결 수명·바이트 수
sudo gethostlatency-bpfcc      # DNS 조회 지연 분포
sudo execsnoop-bpfcc           # 어떤 프로세스가 어떤 명령을 실행했나
eBPF는 "왜 갑자기 99분위 응답이 늘었지?", "어떤 클라이언트가 재전송을 유발하지?" 같은 질문에 강력합니다. Cilium·Pixie·Coroot 같은 도구들이 이를 활용해 관측·네트워킹·보안을 한 번에 다룹니다.
Service Mesh

모든 마이크로서비스 호출을 가로채는 사이드카

Istio·Linkerd·Consul은 각 파드 옆에 사이드카 프록시(주로 Envoy)를 배치합니다. 앱의 모든 인바운드/아웃바운드 트래픽이 이 프록시를 거치므로 — 앱 코드를 바꾸지 않고 — 다음을 얻습니다:

  • 자동 mTLS: 서비스 간 통신이 자동 암호화 + 신원 검증
  • 관측: 모든 호출의 RPS·p99·에러율을 자동 메트릭으로
  • 분산 트레이싱: trace ID 자동 전파, span 자동 생성
  • 트래픽 정책: 카나리 배포 5%·재시도·서킷브레이커·rate limit를 YAML 한 줄로
  • 인가: "service-a만 service-b를 호출 가능" 같은 ACL
반대로 메시는 비용입니다 — 사이드카가 메모리·CPU·지연을 더합니다. 규모와 복잡성을 보고 도입을 결정하세요. eBPF 기반(Cilium Service Mesh)은 사이드카 없이 비슷한 기능을 제공합니다.
Tracing

요청 하나의 인생 — 분산 트레이싱

모놀리식이 마이크로서비스로 쪼개진 뒤 "이 요청이 왜 느리지?"는 경계를 가로지르는 질문이 됐습니다. 해법은 trace ID 전파 — 각 호출이 같은 trace ID + 부모 span ID를 가지면, 끝에서 한 그래프로 재조립할 수 있습니다. 

# W3C Trace Context (HTTP 헤더로 자동 전파)
traceparent: 00-{trace-id 32hex}-{span-id 16hex}-01
tracestate: rojo=00f067aa0ba902b7,congo=t61rcWkgMzE

OpenTelemetry가 사실상 표준입니다. SDK가 자동으로 헤더 전파·span 생성을 처리하고, 백엔드는 Jaeger·Tempo·Honeycomb 등 선택. 중요 지표 4가지(USE·RED·Golden Signals)를 SLO와 연결해 알람으로 잇습니다.

RED
Rate · Errors · Duration — 서비스 단위 외부 관점
USE
Utilization · Saturation · Errors — 리소스(CPU·메모리·디스크) 관점
Golden Signals
Latency · Traffic · Errors · Saturation — Google SRE
실무 워크플로

장애가 났을 때 — 추적 순서

  1. 지표·대시보드로 어떤 서비스가 비정상인지 추정 (RED, Golden Signals)
  2. 트레이스로 그 서비스 안 어느 호출이 느린지/실패하는지 추적
  3. 로그로 해당 트레이스 ID의 컨텍스트를 확인 — 에러 메시지·스택
  4. 네트워크 의심 시 eBPF로 커널 관점 검증 — 재전송·연결 거절·DNS 지연
  5. 마지막 결정타로 tcpdump·Wireshark — 패킷 그 자체
"지표 → 트레이스 → 로그 → 커널 이벤트 → 패킷"의 순서는 가설을 좁히고, 비싼 도구는 늦게라는 원칙의 응용입니다. 평소 관측 인프라를 갖춰 두면 새벽 3시에 길을 잃지 않습니다.
