VPN — 다른 네트워크를 내 네트워크처럼

VPN의 본질은 두 가지입니다 — 암호화로 도청을 막고, 캡슐화(터널링)로 마치 두 끝점이 같은 네트워크에 붙어 있는 것처럼 만들기. 프로토콜이 여럿인 이유는 누가, 어느 계층에서, 어떤 트레이드오프로 그 두 가지를 푸는지가 다르기 때문입니다.

캡슐화

패킷을 다른 패킷 안에 넣기

원본 IP 패킷을 통째로 암호화한 뒤, 새로운 "운반용" IP 헤더를 붙여 인터넷을 가로지릅니다. 도착해서 헤더를 벗기고 복호화하면 원본이 되살아납니다. 이 과정에서 원본의 출발지·도착지조차 외부에는 보이지 않습니다.

사용 모델

두 가지 배포

Site-to-Site

  • 두 사무실/데이터센터의 라우터끼리 상시 터널
  • 내부망 간 통신을 인터넷 우회로 안전하게
  • 주로 IPsec tunnel mode

Remote Access (Client VPN)

  • 개인 단말에서 사내망으로
  • WireGuard, OpenVPN, Cisco AnyConnect 등
  • 요즘은 Zero Trust로 대체되는 추세 (각 앱이 인증)
IPsec

전통의 표준

IPsec은 단일 프로토콜이 아니라 모음입니다.

IKE (v2)
키 교환·인증·정책 협상. UDP/500, NAT 통과 시 UDP/4500.
ESP
실제 데이터 보호. 암호화 + 무결성 (요즘은 AES-GCM 같은 AEAD).
AH
무결성만 (암호화 없음). 거의 안 씀.
Transport mode
원본 IP 헤더 유지, payload만 보호. 호스트-호스트.
Tunnel mode
원본 IP까지 통째로 캡슐화. 게이트웨이-게이트웨이.
IPsec은 강력하지만 구성이 복잡하고 옵션이 매우 많아 오설정이 잦습니다. NAT·방화벽과의 상호작용도 까다롭습니다 (그래서 NAT-T가 별도 표준으로 등장).
WireGuard

현대적 단순함

WireGuard는 "필요한 것만 남기자"는 철학으로 다시 짠 VPN입니다. 암호 스위트는 단 하나 — Curve25519(키 교환), ChaCha20-Poly1305(AEAD), BLAKE2(해시), SipHash(테이블). 커널 코드 약 4,000줄로 IPsec의 수십만 줄 대비 감사가 가능한 크기.

Tor

onion routing — 신원 자체를 가리기

VPN은 "내가 어디로 가는지를 ISP에게 가린다"이지만, Tor는 한 단계 더 나아가 "도착지도 출발지를 모르게" 합니다. 패킷을 3개의 양파 껍질로 감싸 — 각 노드는 자기 다음 노드만 알 수 있도록 — 보냅니다.

  1. Entry (Guard): 사용자 IP를 안다. 다음이 어디인지는 모름.
  2. Middle: 사용자도, 도착지도 모름. Entry와 Exit만 안다.
  3. Exit: 도착지를 안다. 사용자 IP는 모름. (이 노드가 평문을 본다 — HTTPS 필수)
Exit 노드는 트래픽을 그대로 인터넷에 내보냅니다. HTTPS가 아니면 도청 가능합니다. 또한 글로벌 관찰자가 Entry·Exit 양쪽 트래픽 패턴을 비교하면 (correlation) 익명성이 깨질 수 있습니다.
한계

VPN이 풀지 못하는 것

  • 엔드포인트 보안: 단말이 감염되면 VPN으로 사내망에 침투 경로 제공
  • 지나친 신뢰: "VPN으로 접속했으니 사내인 척" 가정 → 측면 이동 쉬움 (Zero Trust가 등장한 이유)
  • DNS 누수: VPN 활성 중에도 OS가 평소 리졸버에 묻는 경우
  • 운영자 신뢰: 상용 VPN 사업자는 ISP를 ISP로 대체하는 효과 — 로그 정책 검증 어려움
