OAuth · OIDC · JWT · mTLS — 누구냐, 무엇을 할 수 있느냐

TLS는 채널을 안전하게 만들지만, "이 요청을 보낸 게 진짜 누구인지", "그 사람이 무엇을 할 권한이 있는지"는 별도의 문제입니다. OAuth는 비밀번호 공유 없이 권한을 위임하기 위해, OIDC는 "신원" 자체를 다루기 위해, JWT는 그 정보를 운반하는 컨테이너로, mTLS는 서비스 간 신원으로 — 각자 다른 층의 문제를 풉니다.

왜 OAuth가 만들어졌나

비밀번호 위임의 시대를 끝내려고

과거에는 외부 서비스가 사용자의 Gmail에 접근하려면 사용자에게 비밀번호를 요구했습니다. 그건 (1) 모든 권한을 통째로 넘기는 것이고, (2) 외부 서비스가 비밀번호를 저장해야 했죠. OAuth는 그 대신 좁은 범위의 토큰을 발급해 위임합니다.

OIDC

OAuth + 신원

OAuth만으로는 "이 사용자가 누구인지"를 표준화된 방식으로 알 수 없습니다 (API에 따라 다름). OIDC(OpenID Connect)는 OAuth 위에 ID Token(JWT)을 얹어 "이 토큰의 주인은 sub=12345의 user@example.com"이라는 정보를 표준화합니다. 그래서 로그인 용도에는 OAuth가 아니라 OIDC가 정답입니다. 

// /.well-known/openid-configuration
{
  "issuer": "https://accounts.example.com",
  "authorization_endpoint": ".../authorize",
  "token_endpoint": ".../token",
  "userinfo_endpoint": ".../userinfo",
  "jwks_uri": ".../.well-known/jwks.json",
  "response_types_supported": ["code"],
  "id_token_signing_alg_values_supported": ["RS256"]
}
JWT

토큰의 컨테이너 — 점으로 구분된 세 조각

JWT는 header.payload.signature의 Base64URL 인코딩 세 부분으로 구성됩니다. 서명은 발급자의 비밀키/사설키로 만들어지고, 검증자는 발급자의 공개키(JWKS)를 받아 확인합니다. 

// header
{ "alg": "RS256", "kid": "abc123", "typ": "JWT" }

// payload (claims)
{
  "iss": "https://accounts.example.com",
  "sub": "12345",
  "aud": "client-id",
  "exp": 1735689600,
  "iat": 1735686000,
  "scope": "openid profile email"
}

// signature
RSA-SHA256( base64url(header) + "." + base64url(payload), private_key )
주의: alg: none 수락 금지, alg을 서버가 강제할 것, aud·iss 검증 필수, 만료 짧게(보통 5~15분), 회수는 introspection 또는 짧은 만료로 달성.

mTLS

서비스 간 신원 — 인증서를 양쪽이

보통의 TLS는 클라이언트가 서버의 인증서를 검증하는 단방향입니다. mTLS는 서버도 클라이언트의 인증서를 검증합니다 — 즉, 인증서 자체가 신원이 됩니다. Bearer 토큰처럼 "탈취 후 어디서나 쓸 수 있는" 약점이 없어 서비스 메시·기관 간 통신에 강합니다.

SPIFFE / SPIRE

클라우드 네이티브 환경에서 워크로드(파드, VM)에 자동으로 인증서를 부여하기 위한 표준. spiffe://example.org/ns/team-a/sa/billing 같은 SPIFFE ID가 워크로드 신원. SPIRE는 그 ID 발급/회전을 자동화한 런타임. Istio 같은 서비스 메시가 내부적으로 활용합니다.

Zero Trust

"내부망이라 안전"이라는 가정을 버리기

전통적 보안은 perimeter — "방화벽 안쪽은 신뢰"였습니다. 그러나 클라우드·SaaS·재택근무가 일상이 된 지금 그 안팎의 구분 자체가 무의미해졌습니다. Zero Trust는 다음 원칙입니다.

  • 매 요청마다 검증: 누가, 어떤 기기에서, 어떤 컨텍스트로 — 매번.
  • 최소 권한: scope·role·time-bound 토큰.
  • 네트워크 위치는 신뢰 근거가 아님: VPN으로 들어왔어도 똑같이 검증.
  • 지속적 평가: 세션 도중에도 위험 신호가 보이면 권한을 깎거나 재인증 요구.
  • BeyondCorp(Google), Cloudflare Access, Tailscale 등이 구체적 구현 예.
함정

자주 실수하는 곳

  • localStorage에 access token 저장 (XSS에 통째로 노출) → HttpOnly 쿠키 + CSRF 방어 또는 PKCE 흐름 권장
  • JWT를 그대로 신뢰 — 서명·iss·aud·exp 모두 검증해야
  • refresh token rotation 없이 영구 사용 — 탈취 시 무한 갱신
  • redirect URI 화이트리스트 누락 → open redirect로 코드 탈취
  • scope를 너무 넓게 요청 — 사용자가 동의해도 위험은 그대로
