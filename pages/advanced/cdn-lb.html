CDN · Anycast · 로드밸런싱 — 트래픽을 길들이는 법

지구 반대편 사용자에게 250ms RTT가 걸리는 원본 서버를 그대로 보여줄 수는 없습니다. 해결은 두 단계입니다 — (1) 콘텐츠를 사용자 근처로 복제해 두고(CDN edge), (2) 사용자를 가장 가까운 복제본으로 안내합니다(Anycast 또는 DNS 지오 라우팅). 그 안쪽에서는 또 다른 안내가 필요합니다 — 한 데이터센터 안에서 어떤 서버로 보낼지(로드밸런서).

CDN 기본

왜 거리가 문제인가

빛은 진공에서 300,000 km/s. 한국–미국 동부 약 11,000km × 2(왕복) × 광섬유 굴절률 ≈ 110ms. 여기에 큐잉·라우팅·TLS handshake가 더해지면 250ms는 우습습니다. TCP는 핸드셰이크에 1.5 RTT가 필요하니, 그것만으로도 0.4초가 사라집니다.

캐시

엣지 캐시의 구조

CDN 엣지는 origin의 정적 자원을 일정 시간(TTL) 저장합니다. 같은 자원을 요청하는 다음 사용자는 origin까지 가지 않습니다.

같은 엣지에 동시에 같은 미스가 몰리면 origin이 폭격당할 수 있어 request coalescing(같은 요청 하나로 합쳐 origin 한 번만 호출) 또는 origin shielding(엣지보다 한 단 위의 캐시 층)을 둡니다.

근접 라우팅

가까운 엣지로 보내는 두 가지 방법

DNS 기반 지오 라우팅

  • 리졸버의 IP로 사용자 지역을 추정해 가까운 엣지 IP를 응답
  • 구현 단순, 운영 통제 쉬움
  • 리졸버가 사용자와 멀리 있으면 부정확 (해소: ECS — EDNS Client Subnet)

Anycast

  • 전 세계 엣지가 같은 IP를 BGP로 광고
  • 네트워크 자체가 AS hop이 짧은 곳으로 라우팅 → 자동 근접 선택
  • DDoS 흡수, BGP 변동에 자동 적응. 단점: 세션 중 다른 엣지로 옮겨가면 문제
실무에서는 보통 둘을 결합합니다 — DNS로 거시적 지역 분배 + 각 지역에서 Anycast로 미시 분배. Cloudflare는 Anycast 비중이 크고, 전통 CDN(예: Akamai)은 DNS 기반에 더 의존합니다.

알고리즘

어떤 서버로 보낼지 정하는 법

round-robin
순서대로 돌아가며 보낸다. 가장 단순, 가중치 옵션 있음. 서버 부하 차이는 못 봄.
least-connections
현재 연결이 가장 적은 서버로. 처리 시간이 다양한 워크로드에 강함.
least-response-time
최근 응답이 빠른 쪽 선호. 느려진 서버를 자동 배제하는 효과.
consistent hashing
요청의 키(예: 사용자 ID)를 해시해 항상 같은 서버로. 서버 수가 바뀌어도 대부분의 키는 같은 서버에 머묾 — 캐시 적중률 보존.
maglev / rendezvous
consistent hash의 진화형. Google Maglev은 충돌 해소 테이블을 미리 만들어 두고 lookup이 O(1).
EWMA
지수 이동 평균으로 서버별 응답시간을 추적하며 더 빠른 곳에 가중치.
P2C (Power of Two Choices)
무작위 두 서버 중 부하 적은 쪽 선택. 거의 균등 분산을 매우 저렴하게 달성.
health check
주기적인 핑/HTTP probe로 비정상 서버 제외. fail-open vs fail-close 정책 선택.
상태

sticky session — 같은 사용자를 같은 서버로

세션 상태가 서버 로컬에 있으면, 다음 요청도 같은 서버로 보내야 합니다. 쿠키 주입, 클라이언트 IP 해시, consistent hashing 등으로 구현하지만, 결국 서버가 죽으면 그 세션도 잃습니다. 가능하면 세션을 외부 저장소(Redis 등)에 두는 게 정답입니다.

실패

로드밸런서가 안 풀어 주는 문제들

  • 썬더링 허드: 캐시 무효화 직후 모든 사용자가 origin으로 몰림 → coalescing, jittered TTL
  • retry 폭풍: 5xx에 대해 클라이언트 모두가 동시에 재시도 → exponential backoff + jitter
  • 흐름 보존: 서버 추가/제거 시 진행 중 연결을 어떻게 다룰지 — connection draining
  • 비대칭 라우팅: 트래픽이 다른 경로로 돌아오는 경우 stateful 방화벽이 끊을 수 있음
