04 · Transport (L4)

전송 계층 — 정확성과 속도 사이의 두 갈래

IP는 "보내 본다"까지만 책임집니다. 그 위에서 누가 받아야 하는지(포트), 잃어버린 건 어떻게 할지, 너무 많이 보낼 때는 어떻게 줄일지를 정하는 게 전송 계층입니다. TCP는 정확성을, UDP는 단순함과 저지연을 택합니다.

TCP

연결 지향 · 신뢰성 · 순서 보장

TCP는 통신 전에 양 끝점이 서로의 상태를 일치시키는 3-way handshake를 거칩니다. 이후 보내는 각 바이트에 시퀀스 번호가 붙고, 받는 쪽은 ACK으로 "여기까지 잘 받았다"고 알립니다. 손실되거나 순서가 뒤집히면 재전송·재배치를 통해 위 계층에는 깨끗한 바이트 스트림으로 올려줍니다.

핵심 메커니즘

  • 3-way handshake: SYN → SYN+ACK → ACK. 양쪽이 서로의 ISN(초기 시퀀스 번호)을 교환.
  • 흐름 제어: 수신 윈도우(rwnd)로 "지금 이만큼만 받을 수 있다"를 알려 송신자를 조절.
  • 혼잡 제어: 네트워크 상태를 추정해 송신량을 늘렸다(slow start) 줄였다(congestion avoidance).
  • 재전송: 타이머 만료 또는 중복 ACK 감지 시 재전송 (Fast Retransmit).
  • 4-way FIN: 종료도 양쪽이 각각 종료 신호를 보낸다 (half-close 가능).

TCP가 적합한 곳

  • 파일 전송, 웹 페이지, 이메일, DB 트랜잭션 — 한 바이트도 잃으면 안 되는 경우
  • 긴 시간 양방향 스트림 (SSH, 메시징)
  • 상위 앱이 신뢰성 코드를 직접 안 짜고 싶을 때 (대부분)

TCP가 불리한 곳

  • 연결 수립의 RTT 비용이 큰 짧은 트랜잭션 (DNS 단발 질의)
  • 지연이 손실보다 더 치명적인 실시간 (게임, 음성)
  • 한 스트림 손실이 다른 스트림까지 막는 "Head-of-Line 블로킹" (HTTP/2 over TCP의 단점, HTTP/3로 해소)
  • 보안: SYN flood — 수많은 SYN으로 서버의 half-open 큐 채워 정상 사용자 거부
UDP

비연결 · 데이터그램

UDP는 헤더가 8바이트밖에 안 됩니다. 출발지 포트, 도착지 포트, 길이, 체크섬. 그게 전부입니다. 핸드셰이크도, 재전송도, 순서 정렬도 없습니다. "보내고 잊는다(fire-and-forget)"는 단순함 덕에 빠르지만, 그 모든 보장은 앱이 직접 만들어야 합니다.

UDP가 적합한 곳

  • DNS 질의: 1패킷 요청 → 1패킷 응답, 핸드셰이크가 부담
  • VoIP·게임·실시간 영상: 약간의 손실이 큰 지연보다 낫다
  • QUIC(HTTP/3): UDP 위에 신뢰성·암호화를 새로 설계해서 얹음
  • 대량 텔레메트리·로그 (statsd 등) — 일부 손실 허용

UDP가 불리한 곳

  • 장기 양방향 정확한 데이터 전송 (TCP가 만들어 둔 모든 것을 다시 짜야 함)
  • NAT/방화벽 통과: 상태가 없으므로 일부 NAT가 빨리 닫아버림
  • 보안: 반사·증폭 공격(DNS·NTP·memcached) — 작은 요청에 큰 응답

진화

QUIC — TCP의 좋은 점만, UDP 위에서

HTTP/3가 사용하는 QUIC은 UDP 위에서 동작하지만 신뢰성·순서·혼잡 제어·암호화(TLS 1.3 통합)를 모두 갖춥니다. TCP의 굳어진 한계(예: HoL 블로킹, 연결 마이그레이션 불가, 미들박스 고착)를 우회하기 위한 설계입니다.

QUIC은 "왜 굳이 UDP를 골랐을까?"라는 질문의 좋은 답입니다. TCP에 새 기능을 추가하면 전 세계 미들박스가 거부하는 경우가 많아, OS 커널 밖에서 진화 가능한 새 전송을 UDP 위에 얹은 것이죠.
