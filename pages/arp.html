02 · Data Link (L2)

ARP — 같은 LAN에서 "그 IP의 MAC"을 찾기

IP는 L3 주소지만, 실제로 같은 링크에서 프레임을 보내려면 L2의 MAC 주소가 있어야 합니다. ARP(Address Resolution Protocol)는 IP → MAC 매핑을 찾는 작은 프로토콜입니다. 단순한 만큼 인증이 없어 보안에는 약점이 됩니다.

왜 필요한가

스위치는 MAC으로만 전달한다

같은 스위치에 붙은 두 기기 A와 B가 있다고 합시다. A는 B의 IP는 알지만, 스위치는 이더넷 프레임을 MAC 주소로 전달합니다. 그래서 A는 먼저 "192.168.0.20을 가진 사람, 너의 MAC이 뭐야?"를 LAN 전체에 묻습니다.

  1. ARP Request (브로드캐스트): A → ff:ff:ff:ff:ff:ff — "Who has 192.168.0.20? Tell 192.168.0.10."
  2. 스위치는 브로드캐스트이므로 같은 VLAN의 모든 포트로 복제 전달합니다.
  3. ARP Reply (유니캐스트): B → A — "192.168.0.20 is at bb:bb:bb:bb:bb:bb."
  4. A는 ARP 캐시에 저장하고, 다음 통신부터는 즉시 MAC을 알아봅니다.
트레이드오프

강점과 약점

적합한 상황

  • 같은 L2 도메인 안의 빠르고 단순한 주소 해석
  • 캐시 기반이라 반복 통신에서 거의 오버헤드 없음
  • 구현이 단순해 모든 OS·NIC에 기본 탑재

약점

  • 인증이 없다: 누구든 "내가 그 IP다"라고 응답 가능 → ARP 스푸핑
  • 브로드캐스트라 큰 LAN에서는 트래픽 비용
  • L3를 넘지 못함 (라우터 너머의 IP는 ARP로 못 찾음 — 라우터의 MAC을 찾음)
미리보기

ARP 스푸핑 (Gratuitous ARP)

공격자 X가 묻지도 않았는데 "192.168.0.1(게이트웨이)은 내 MAC이야"라고 반복적으로 브로드캐스트하면, A의 ARP 캐시가 덮어쓰이면서 외부로 나가는 모든 트래픽이 X를 거쳐 가게 됩니다. 자세한 시각화는 보안 페이지에서 다룹니다.

ARP는 "마지막에 들은 응답을 믿는" 단순한 모델입니다. 그래서 신뢰할 수 없는 네트워크(공용 Wi-Fi 등)에서는 상위 계층 암호화(TLS)와 인증이 필수입니다.
