# 시나리오 14 walkthrough: 소켓 (LISTEN / ESTABLISHED / TIME_WAIT)

> 랩 시나리오 `scenarios/14-sockets.sh`를 실제로 돌려 캡처한 결과를 한 줄씩 해설한다.
> 교재 매핑: `next/content/tcp-udp.mdx`(소켓 상태/TIME_WAIT), `advanced/numbers.mdx`(소켓·FD)
>
> 재현: `cd lab && LAB_STEP=1 ./labctl run 14-sockets`

## 등장인물

| 역할 | 주소 | 비고 |
|---|---|---|
| server | `10.80.0.1` | `iperf3 -s`로 5201 리슨 |
| client | `10.80.0.2` | `iperf3 -P 4`로 병렬 연결 |

소켓은 **(프로토콜, 로컬IP:포트, 원격IP:포트)** 4-tuple로 연결을 식별한다. 이 한 줄이
"왜 포트 하나로 수많은 연결을 받을 수 있는가"를 설명한다. `ss`로 소켓의 상태를 직접 들여다본다.

## ① LISTEN: 연결을 받을 "귀"

```text
[server]$ ss -tlnp
State  Recv-Q Send-Q Local Address:Port  Peer  Process
LISTEN 0      4096   *:5201              *:*   users:(("iperf3",pid=23,fd=3))
```

- `LISTEN` 소켓은 아직 상대가 없다. **로컬 포트(5201)만** 정해져 있고 Peer는 `*:*`(아무나)이다.
  "연결을 받을 귀"로 열려 있는 상태.
- `fd=3`에 주목. 소켓은 프로세스 입장에서 **파일 디스크립터(FD)** 다. 그래서 "동시 연결 수"는 곧
  "열린 FD 수"이고, `ulimit -n`(FD 한계)에 부딪힌다. numbers.mdx의 "100만 동시 연결" 얘기가 이 지점이다.
- `Recv-Q`의 `4096`은 accept 대기 큐(backlog) 크기다.

## ② ESTABLISHED: 한 포트에 여러 소켓 (server 시점)

```text
[server]$ ss -tan | grep 5201
LISTEN 0      4096   10.80.0.1:5201   *:*
ESTAB  ...    0      10.80.0.1:5201   10.80.0.2:55184
ESTAB  ...    0      10.80.0.1:5201   10.80.0.2:55158
ESTAB  ...    0      10.80.0.1:5201   10.80.0.2:55180
ESTAB  ...    0      10.80.0.1:5201   10.80.0.2:55192
ESTAB  ...    0      10.80.0.1:5201   10.80.0.2:55174
```

- `iperf3 -P 4`가 동시에 여러 연결을 맺었다(데이터 4 + 제어 1 = 5개).
- 결정적 관찰: **Local 주소가 다섯 줄 모두 `10.80.0.1:5201`로 똑같다.** 그런데 별개의 소켓이다.
- 무엇이 이들을 구분하나? **Peer 포트**다(55184, 55158, 55180, 55192, 55174). 4-tuple 중 원격 포트가
  다르므로 커널은 각각을 다른 연결로 안다. **그래서 listening 포트 하나가 수만 개의 연결을 받아낸다.**
  "포트가 부족하다"는 흔한 오해와 달리, 서버의 listening 포트는 하나면 충분하다.

> 주소가 `[::ffff:10.80.0.1]` 형태로 보이면 IPv4-mapped IPv6다. iperf3가 IPv6 듀얼스택 소켓으로
> 받아서 그렇고, 의미는 `10.80.0.1`과 같다.

## ③ 같은 연결을 client 시점에서

```text
[client]$ ss -tan | grep 5201
ESTAB 0 ... 10.80.0.2:55180  10.80.0.1:5201
ESTAB 0 ... 10.80.0.2:55192  10.80.0.1:5201
ESTAB 0 ... 10.80.0.2:55158  10.80.0.1:5201
...
server :5201 ESTABLISHED 소켓 수 = 5
```

- 같은 연결들인데 시점이 뒤집혀 보인다. 클라이언트는 매 연결마다 **임시 포트(ephemeral port)**를
  하나씩 새로 써서(55180, 55192, ...) 같은 `server:5201`로 향한다.
- 그래서 클라이언트 쪽은 "포트 고갈"이 실재하는 문제다. 한 목적지로 많은 연결을 열면 로컬 임시 포트
  범위(보통 ~28k개)를 다 쓸 수 있다. (서버 쪽이 아니라 클라이언트/프록시 쪽 이야기다.)

## ④ TIME_WAIT: 먼저 닫은 쪽에 남는다

```text
server쪽 TIME_WAIT: 0  /  client쪽 TIME_WAIT: 5

[client]$ ss -tan state time-wait | grep 5201
0 0 10.80.0.2:55180  10.80.0.1:5201
0 0 10.80.0.2:55192  10.80.0.1:5201
...
```

- iperf3가 끝나고 연결이 닫히자, **client에 TIME_WAIT 소켓 5개**가 남았다. server는 0이다.
- 차이의 원인: TIME_WAIT은 **능동적으로 먼저 close한 쪽(active close)**에 생긴다. 여기선 클라이언트가
  먼저 닫았다(시나리오 05의 FIN 교환에서 먼저 FIN을 보낸 쪽).
- 왜 바로 안 없애고 남겨두나? 늦게 도착한 옛 패킷이 같은 4-tuple로 맺어질 **다음 연결을 오염시키지
  않도록**, 2MSL(보통 60초 안팎) 동안 그 4-tuple을 잡아둔다.
- 이것이 교재 tcp-udp의 "TIME_WAIT 폭증 -> 포트 고갈"의 실체다. 짧은 연결을 대량으로 여닫는 쪽
  (보통 클라이언트/리버스 프록시)에서 TIME_WAIT 소켓이 임시 포트를 점유해 새 연결을 못 맺는 사고가 난다.

## 한눈 정리

| 상태 | 의미 | 누구에게 |
|---|---|---|
| LISTEN | 연결 받을 귀 (로컬 포트만 정해짐, FD 하나) | 서버 |
| ESTABLISHED | 4-tuple로 식별되는 활성 연결 | 양쪽 |
| TIME_WAIT | 닫은 뒤 2MSL 대기 (옛 패킷 방지) | 먼저 close한 쪽 |

핵심 한 줄: **소켓 = 연결을 식별하는 4-tuple이자 프로세스의 파일 디스크립터.** 그래서
서버는 포트 하나로 무수한 연결을 받고(구분은 원격 IP:포트), 동시 연결 수의 천장은 포트가 아니라
FD 한계이며, 짧은 연결을 남발하는 쪽은 TIME_WAIT으로 임시 포트가 마른다.
