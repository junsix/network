# 시나리오 15 walkthrough: 동시성 모델 (워커 점유 vs 이벤트 기반)

> 랩 시나리오 `scenarios/15-concurrency.sh`를 실제로 돌려 캡처한 결과를 한 줄씩 해설하고,
> 뒤에 개념(소켓 vs 스레드, 동시성 모델, C10K)을 정리한다.
> 교재 매핑: `next/content/advanced/numbers.mdx` (소켓-스레드 비대칭). 시나리오 14의 후속.
>
> 재현: `cd lab && LAB_STEP=1 ./labctl run 15-concurrency`

## 등장인물

| 역할 | 주소/포트 | 동시성 모델 |
|---|---|---|
| server (블로킹) | `10.90.0.1:8080` | nc 루프 = 한 번에 연결 **1개**만 처리 |
| server (이벤트) | `10.90.0.1:8081` | nginx = epoll 이벤트 루프 |
| client | `10.90.0.2` | 정상 요청 + "느린 연결"을 만든다 |

소켓은 싸다(시나리오 14에서 한 포트에 소켓 수천 개를 봤다). 하지만 그 소켓을 **어떤 실행 단위로
처리하느냐**가 동시성을 가른다. 같은 "느린 클라이언트"를 양쪽 서버에 붙여 그 차이를 잰다.

## ① 기준선: 둘 다 즉시 응답

```text
블로킹(8080):  HTTP=200  소요=0.0018s
이벤트(8081):  HTTP=200  소요=0.0006s
```

평상시엔 둘 다 1~2ms로 멀쩡하다. 차이는 "느린 연결"이 끼어들 때 드러난다.

## ② 느린 연결이 끼어들면

`sleep 8 | nc ...`로 각 서버에 **요청을 끝내지 않고 8초간 붙잡는** 연결을 하나씩 만들었다.
그 상태에서 정상 요청을 다시 던진다.

```text
[블로킹 8080] 점유 중 정상 요청:  HTTP=000  소요=3.0048s   <- 타임아웃(막힘)
[이벤트 8081] 점유 중 정상 요청:  HTTP=200  소요=0.0006s   <- 영향 없음
```

- **블로킹 서버(8080)**: 동시성이 1이라, 느린 연결 하나가 **유일한 워커를 붙잡는다.** 그동안 새 요청은
  accept조차 못 돼(또는 처리 못 돼) `HTTP=000`(3초 타임아웃)으로 실패한다. 서버는 살아 있는데
  "일손이 없어" 못 받는 상태다.
- **이벤트 서버(8081)**: 느린 연결은 epoll이 감시하는 **항목 하나**일 뿐, 워커(스레드)를 붙잡지 않는다.
  그래서 다른 요청은 그대로 0.6ms에 처리된다.

같은 느린 연결, 정반대 결과. 이 한 줄(`000` vs `200`)이 동시성 모델의 차이 전부를 보여준다.

# 개념 정리

## 소켓 != 스레드 (두 자원의 비용 비대칭)

| | 소켓(연결) | 스레드/워커 |
|---|---|---|
| 정체 | 4-tuple로 식별되는 커널 객체 + FD | 실행 단위(스택 + 스케줄러 대상) |
| 비용 | idle 연결당 **~수십 KB** (numbers.mdx) | 스레드당 **~MB 스택** + 컨텍스트 스위치 |
| 한계 | FD 한계 / 포트(클라이언트측) | CPU 코어 수 × 스케줄링 비용 |

핵심: **연결을 여는 것(소켓)은 싸지만, 연결마다 스레드를 붙이는 것은 비싸다.** 그래서 "연결 = 스레드"
모델은 연결 수가 늘면 스레드 비용에 먼저 무너진다. 이 비대칭이 아래 모든 모델이 갈라지는 이유다.

## 동시성 모델 4가지

| 모델 | 한 연결을 처리하는 단위 | 느린 연결에 약한가? | 대표 |
|---|---|---|---|
| thread-per-connection | OS 스레드 1개/연결 | 매우 (스레드 풀 고갈) | 옛 Apache(prefork), 단순 블로킹 서버 |
| thread pool (N workers) | 풀의 워커 1개/요청 | 느린 연결 N개면 고갈 | Tomcat 기본, 많은 WAS |
| event loop (epoll/async) | 워커 없음, 이벤트 콜백 | 거의 안 (수천~수만 동시) | nginx, Node.js, Netty, Go net/http 내부 |
| coroutine / 경량 스레드 | 유저공간 경량 단위 | 거의 안 (스택 ~KB) | Go goroutine, Java Virtual Thread |

이 랩의 블로킹 서버는 **thread pool에서 N=1**인 극단이다. 그래서 느린 연결 **하나**로 재현됐지만,
N=200인 실서버도 느린 연결 **200개**면 똑같이 고갈된다(이게 slowloris 공격의 원리다).

## C10K, 그리고 C10M

- **C10K**(2000년대 초): "한 서버가 동시 1만 연결을 감당할 수 있나"라는 문제. thread-per-connection으론
  불가능했다(1만 스레드 × MB = GB + 스케줄링 붕괴). 해법이 **이벤트 기반(epoll/kqueue)**이었고,
  그래서 nginx/Node가 떴다.
- 소켓 비용 자체는 작으므로(numbers.mdx: idle 연결 ~수십 KB), 메모리만 보면 **C1M(동시 100만)**도
  가능하다. 병목은 소켓이 아니라 **"연결을 처리하는 실행 단위"** 였다는 게 C10K의 교훈이다.
- 최근 Go goroutine / Java Virtual Thread는 "블로킹 코드의 단순함"과 "이벤트 루프의 확장성"을 동시에
  노린다. 경량 스레드 스택이 ~KB라 수십만 개를 띄울 수 있다.

## 실무로 읽기

- **타임아웃은 필수다.** 느린/죽은 연결이 워커를 무한정 붙잡지 못하도록 read/write/idle 타임아웃을 건다.
  (이 랩에서 블로킹 서버가 막힌 건 타임아웃이 없어서다.)
- **느린 클라이언트는 공격이 된다.** slowloris는 요청을 일부러 천천히 보내 워커를 점유한다.
  thread-per-request 서버 앞에 **이벤트 기반 리버스 프록시(nginx)**를 두는 게 표준 방어다(느린 연결을
  프록시가 흡수하고, 완성된 요청만 백엔드로 넘긴다. 시나리오 11과 연결된다).
- **"동시 연결 수"의 천장**은 보통 소켓/FD가 아니라 **동시성 모델**이다. 스레드풀 크기, 이벤트 루프
  여부를 먼저 본다.

## 한눈 정리

| | 블로킹/워커 점유 | 이벤트 기반 |
|---|---|---|
| 연결당 비용 | 스레드 1개(비쌈) | epoll 항목 1개(쌈) |
| 느린 연결 | 워커 점유 -> 다른 요청 막힘 | 영향 거의 없음 |
| 확장 한계 | 스레드 수(C10K 벽) | 소켓/FD (훨씬 높음) |
| 단순함 | 코드가 직관적(블로킹) | 콜백/async (복잡) 또는 경량스레드로 절충 |

소켓은 싸고 스레드는 비싸다. 이 한 문장이 thread-per-connection에서 이벤트 루프와 goroutine으로
이어진 서버 아키텍처 진화의 전부다. 랩에서 본 `000` vs `200`이 그 진화가 왜 필요했는지를 보여준다.
