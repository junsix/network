> 랩 시나리오 `scenarios/11-l7-lb.sh`를 실제로 돌려 캡처한 결과를 한 줄씩 해설한다.

>
> 재현: `cd lab && LAB_STEP=1 ./labctl run 11-l7-lb`

## 등장인물

```
client (10.50.0.2)  --->  LB / nginx (10.50.0.1:80)  --+--> backendA (10.50.0.11:8080)
                                                       +--> backendB (10.50.0.12:8080)
```

| 역할 | 주소 | 비고 |
|---|---|---|
| client | `10.50.0.2` | LB만 안다 |
| LB | `10.50.0.1:80` | nginx 리버스 프록시. upstream 라운드로빈 |
| backendA | `10.50.0.11:8080` | "backend-A"를 반환 |
| backendB | `10.50.0.12:8080` | "backend-B"를 반환 |

클라이언트는 **입구(LB) 하나만** 안다. 백엔드 두 대는 LB 뒤에 숨어 있다. LB가 HTTP 요청을 받아(L7)
어느 백엔드로 보낼지 정한다.

LB의 nginx 설정 핵심:

```nginx
upstream backends {
  server 10.50.0.11:8080;
  server 10.50.0.12:8080;
}
server {
  listen 80;
  location / {
    proxy_pass http://backends;
    proxy_set_header X-Forwarded-For $remote_addr;
  }
}
```

## ① 한 입구로 6번 요청하면 두 백엔드로 갈린다

```text
[client]$ for i in $(seq 6); do curl -s http://10.50.0.1/; done
      backend-A
      backend-B
      backend-A
      backend-B
      backend-A
      backend-B

백엔드별 응답 수 -> backend-A: 3 / backend-B: 3
```

- 클라이언트는 매번 **같은 주소 `http://10.50.0.1/`** 로 요청했다. 그런데 응답은 A, B가 번갈아 온다.
- nginx upstream의 기본 정책이 **라운드로빈**이라, 들어오는 요청을 backendA -> backendB -> backendA ...
  순서로 분배했다. 6번 요청이 정확히 3:3으로 갈렸다.
- 클라이언트 입장에선 "주소 하나"가 사실은 뒤의 여러 대로 펼쳐져 있다는 걸 모른다. 이것이 로드밸런서의
  기본 역할: **하나의 가상 입구 뒤로 용량을 수평 확장**한다.

## ② L7의 증거: 연결이 둘로 나뉜다

```text
client -> LB 연결 (목적지 :80):
      10.50.0.2.40010 > 10.50.0.1.80:   Flags [S], ...
      10.50.0.2.40026 > 10.50.0.1.80:   Flags [S], ...

LB -> backend 연결 (목적지 :8080, 출발지는 LB 10.50.0.1):
      10.50.0.1.50392 > 10.50.0.11.8080: Flags [S], ...
      10.50.0.1.46770 > 10.50.0.12.8080: Flags [S], ...
```

- SYN(연결 시작)만 떠 보면 연결이 **두 종류**다.
  - `10.50.0.2 > 10.50.0.1:80`: 클라이언트가 LB에 맺는 연결.
  - `10.50.0.1 > 10.50.0.11:8080`, `10.50.0.1 > 10.50.0.12:8080`: **LB가 백엔드에 새로 맺는 연결.**
- 핵심은 두 번째 묶음의 **출발지가 `10.50.0.1`(LB)** 이라는 점이다. 클라이언트가 아니다.
- 즉 LB는 클라이언트의 TCP를 **자기 선에서 끝내고(terminate)**, 백엔드로는 **완전히 새 TCP를 연다.**
  요청 두 개가 한 줄로 통과하는 게 아니라, LB에서 끊고 다시 잇는다. 이것이 "L7 프록시"의 정의다.

## ③ 그래서 X-Forwarded-For가 필요하다

- 백엔드 입장에서 보면 모든 요청의 출발지가 `10.50.0.1`(LB)이다. 진짜 클라이언트(`10.50.0.2`)의 IP가
  TCP 헤더에는 남지 않는다(LB가 연결을 다시 맺었으니까).
- 그래서 LB는 HTTP 헤더에 `X-Forwarded-For: 10.50.0.2`를 넣어 **원래 클라이언트 IP를 애플리케이션
  계층으로 전달**한다. 백엔드 로그/접근제어가 진짜 클라이언트를 알 수 있는 건 이 헤더 덕분이다.

## L4 LB와의 차이 (한눈 정리)

| | L4 로드밸런서 (NAT/DSR) | L7 로드밸런서 (리버스 프록시) |
|---|---|---|
| 보는 계층 | IP/포트 (TCP) | HTTP 내용 (경로, Host, 쿠키...) |
| 연결 | 한 연결을 백엔드로 전달 | 끊고 새로 맺음 (이 랩에서 본 것) |
| 백엔드가 보는 출발지 | 클라이언트 IP(구성따라) | LB IP (+ X-Forwarded-For) |
| 할 수 있는 일 | 빠른 분산 | 경로 기반 라우팅, TLS 종료, 캐싱, 재시도 |

| 관점 | 내용 |
|---|---|
| 클라이언트 체감 | 주소 하나 (`10.50.0.1`) |
| 실제 | 그 뒤로 backendA/B가 라운드로빈 |
| 연결 구조 | client<->LB 와 LB<->backend 가 별개의 TCP |

로드밸런서는 "입구를 하나로 보이게 하고, 뒤에서 여러 대로 펼치는" 장치다. L7 LB는 거기에 더해
HTTP를 이해하므로 연결을 끊고 다시 맺으며, 그 대가로 원래 클라이언트 정보를 헤더로 따로 전달한다.
