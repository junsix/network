# 시나리오 13 walkthrough: DNS 라운드로빈 (vs L7 로드밸런서)

> 랩 시나리오 `scenarios/13-dns-rr.sh`를 실제로 돌려 캡처한 결과를 한 줄씩 해설한다.
> 교재 매핑: `next/content/dns.mdx` + `advanced/cdn-lb.mdx` (DNS 기반 분산)
>
> 재현: `cd lab && LAB_STEP=1 ./labctl run 13-dns-rr`

## 등장인물

```
client (10.70.0.2)  --(이름 조회)-->  dnssrv (10.70.0.1)
        |                              app.lab.internal -> 10.70.0.11 / .12 / .13
        +--(HTTP 직접)-->  backend1(.11) / backend2(.12) / backend3(.13)
```

| 역할 | 주소 | 비고 |
|---|---|---|
| client | `10.70.0.2` | 이름 `app.lab.internal`만 쓴다 |
| dnssrv | `10.70.0.1` | 한 이름에 A 레코드 3개 |
| backend1/2/3 | `10.70.0.11/12/13` | 각자 자기 이름 반환 (:80) |

시나리오 11(L7 LB)과 같은 "분산"이지만, 분산이 **일어나는 지점**이 다르다. 여기선 프록시가 아니라
**이름 해석(DNS)** 단계에서 갈린다.

## ① 이름 하나에 A 레코드 셋

```text
[client]$ dig +noall +answer @10.70.0.1 app.lab.internal
app.lab.internal.   0   IN   A   10.70.0.12
app.lab.internal.   0   IN   A   10.70.0.11
app.lab.internal.   0   IN   A   10.70.0.13
```

- 한 번의 조회에 **A 레코드 3개**가 한꺼번에 온다. 이것이 "DNS 안에 사는 백엔드 풀"이다.
- 돌아온 순서(12, 11, 13)가 설정 순서(11, 12, 13)와 다르다. DNS 서버가 응답 순서를 섞어 준다.
- 가운데 `0`은 TTL이다. 실제 DNS 라운드로빈에서는 이 TTL이 중요하다. 캐시가 살아 있는 동안은
  같은 답을 재사용하므로, 풀을 바꿔도 TTL이 지나야 반영된다(아래 약점 참고).

## ② 조회마다 순서가 돌아 분산된다

```text
[client]$ for i in $(seq 6); do curl -s http://app.lab.internal/; done
      backend-1
      backend-3
      backend-2
      backend-1
      backend-3
      backend-2
분포 -> b1:2 b2:2 b3:2
```

- 클라이언트의 `/etc/resolv.conf`에 `options rotate`를 줬다. glibc 리졸버가 **조회마다 A 레코드
  순서를 회전**시켜, curl이 매번 다른 첫 주소를 골라 접속한다.
- 6번 요청이 b1, b3, b2를 돌며 정확히 2:2:2로 갈렸다. 프록시 한 대 없이 분산이 됐다.

## ③ 핵심: 클라이언트가 백엔드에 "직접" 붙는다

```text
[8] :80 연결 시작(SYN) 캡처
      10.70.0.2.44360 > 10.70.0.11.80: Flags [S] ...
      10.70.0.2.43894 > 10.70.0.13.80: Flags [S] ...
```

- SYN의 **출발지가 `10.70.0.2`(client), 목적지가 `10.70.0.11`/`10.70.0.13`(백엔드)** 다.
- 즉 클라이언트가 백엔드 IP로 **곧장** TCP를 맺는다. 경로에 프록시가 없다.
- 시나리오 11(L7 LB)과 정반대다. 거기선 클라이언트가 VIP 한 곳(10.50.0.1)에만 붙고, **LB가**
  백엔드로 새 연결을 맺었다. 여기선 LB라는 중간자 자체가 없다.

## L7 LB와의 비교 (이게 이 시나리오의 본론)

| | DNS 라운드로빈 (13) | L7 로드밸런서 (11) |
|---|---|---|
| 분산이 일어나는 곳 | 이름 해석(DNS 응답) | 프록시(매 요청) |
| 경로의 중간자 | 없음 (client -> backend 직접) | LB가 끼어듦 (client -> LB -> backend) |
| 백엔드 IP 노출 | 클라이언트에 그대로 보임 | 숨겨짐 (VIP만) |
| 헬스 체크 | 없음 (죽은 백엔드도 응답에 남음) | 있음 (죽으면 제외) |
| 분산 단위 | 조회 단위(+캐시) | 요청 단위 |
| 콘텐츠 기반 라우팅 | 불가 | 가능(경로/헤더) |

DNS 라운드로빈의 약점이 곧 L7 LB가 존재하는 이유다:

- **헬스 체크 부재**: backend2가 죽어도 DNS는 계속 `.12`를 답한다. 그 답을 받은 클라이언트는
  죽은 서버에 직접 붙어 실패한다. TTL이 만료되고 운영자가 레코드를 빼기 전까진 계속.
- **캐시로 인한 끈적임(stickiness)**: 리졸버/OS가 답을 캐시하면, TTL 동안 같은 백엔드만 때린다.
  분산이 고르지 않을 수 있다.
- **클라이언트가 선택권을 가짐**: 누가 어느 IP를 고를지는 클라이언트(와 그 리졸버) 마음이다.

그래서 둘은 흔히 **함께** 쓴다. DNS로 여러 지역의 LB(VIP)에 거칠게 분산하고(지리적/거대 규모),
각 지역의 L7 LB가 헬스 체크와 정밀 분산을 맡는다. "DNS는 큰 갈래, LB는 세밀한 분배"다.
