# 시나리오 05 walkthrough: TCP 핸드셰이크 vs UDP

> 랩 시나리오 `scenarios/05-tcp-udp.sh`를 실제로 돌려 캡처한 패킷을 한 줄씩 해설한다.
> 교재 매핑: `next/content/tcp-udp.mdx` (L1-04, L4 계층)
>
> 재현: `cd lab && LAB_STEP=1 ./labctl run 05-tcp-udp` (단계/명령마다 Enter 로 멈춤)

## 등장인물

| 역할 | 주소 | 비고 |
|---|---|---|
| client | `172.16.0.2` | 출발 포트는 임시 포트(ephemeral). 실행마다 랜덤 |
| server | `172.16.0.1` | 9000 번 포트 리슨 |
| 스위치 | `br0` | 둘을 잇는 리눅스 브리지(가상 스위치) |

> 주소 안내: `172.16.0.0/12`는 RFC 1918 사설 대역이고, 그나마도 시나리오 실행 중에만
> 존재하던 컨테이너 내부 네임스페이스 IP다. 컨테이너는 `--rm`으로 폐기되므로 실재하지 않는
> 예시값이다. 노출돼도 무방하다.

## tcpdump 한 줄 읽는 법

```
시각  IP  출발지IP.포트 > 목적지IP.포트:  Flags[플래그], seq..., ack..., win..., options[...], length바이트
```

플래그 약어:

| 표기 | 의미 |
|---|---|
| `[S]` | SYN (연결 시작) |
| `[S.]` | SYN + ACK (`.` = ACK) |
| `[.]` | 순수 ACK |
| `[P.]` | PSH + ACK (데이터를 바로 앱에 올림) |
| `[F.]` | FIN + ACK (연결 종료) |

## TCP: 한 줄 메시지에 패킷 8개

`server`에서 `nc -l -p 9000`으로 리슨하고, `client`가 접속해 `hello-over-tcp` 한 줄을 보낸 뒤
캡처를 본 화면이다(프롬프트의 호스트네임은 마스킹).

```text
[3] [TCP] server에서 리슨, client가 접속 후 한 줄 전송
      [client]$ echo "hello-over-tcp" | nc -w1 172.16.0.1 9000
      server 수신 내용: hello-over-tcp

[6] 전체 캡처 비교 (TCP 다수 vs UDP 단발)
      15:05:16.598335 IP 172.16.0.2.38552 > 172.16.0.1.9000: Flags [S], seq 2961618655, win 64240, options [mss 1460,sackOK,TS val 3411137412 ecr 0,nop,wscale 7], length 0
      15:05:16.598365 IP 172.16.0.1.9000 > 172.16.0.2.38552: Flags [S.], seq 1888076871, ack 2961618656, win 65160, options [mss 1460,sackOK,TS val 2939040608 ecr 3411137412,nop,wscale 7], length 0
      15:05:16.598394 IP 172.16.0.2.38552 > 172.16.0.1.9000: Flags [.], ack 1, win 502, options [nop,nop,TS val 3411137412 ecr 2939040608], length 0
      15:05:16.598516 IP 172.16.0.2.38552 > 172.16.0.1.9000: Flags [P.], seq 1:16, ack 1, win 502, options [nop,nop,TS val 3411137412 ecr 2939040608], length 15
      15:05:16.598525 IP 172.16.0.1.9000 > 172.16.0.2.38552: Flags [.], ack 16, win 509, options [nop,nop,TS val 2939040608 ecr 3411137412], length 0
      15:05:17.601516 IP 172.16.0.2.38552 > 172.16.0.1.9000: Flags [F.], seq 16, ack 1, win 502, options [nop,nop,TS val 3411138415 ecr 2939040608], length 0
      15:05:17.601919 IP 172.16.0.1.9000 > 172.16.0.2.38552: Flags [F.], seq 1, ack 17, win 509, options [nop,nop,TS val 2939041612 ecr 3411138415], length 0
      15:05:17.602038 IP 172.16.0.2.38552 > 172.16.0.1.9000: Flags [.], ack 2, win 502, options [nop,nop,TS val 3411138416 ecr 2939041612], length 0
```

### ① SYN: "연결 열자" (client to server)

```text
15:05:16.598335  172.16.0.2.38552 > 172.16.0.1.9000:  Flags [S], seq 2961618655, win 64240,
                 options [mss 1460,sackOK,TS val 3411137412 ecr 0,nop,wscale 7], length 0
```

- `Flags [S]`: SYN. 핸드셰이크 1단계. client가 문을 두드린다.
- `seq 2961618655`: client의 초기 시퀀스 번호(ISN). 보안상 랜덤. 앞으로 보낼 바이트를 여기서부터 센다.
- `win 64240`: 내 수신 버퍼 여유(흐름 제어).
- `options`: `mss 1460`(한 번에 받을 최대 세그먼트), `sackOK`(선택적 ACK), `TS`(RTT 측정용 타임스탬프), `wscale 7`(윈도우를 2^7=128배 확장). 연결 협상값이라 SYN 계열에만 붙는다.
- `length 0`: 데이터 0바이트. 순수하게 연결만 여는 패킷이다.

### ② SYN-ACK: "좋아, 나도 연다" (server to client)

```text
15:05:16.598365  172.16.0.1.9000 > 172.16.0.2.38552:  Flags [S.], seq 1888076871, ack 2961618656, win 65160, ...
```

- `Flags [S.]`: SYN + ACK. 2단계. server가 자기 연결을 열면서(S) 상대 SYN을 받았다고 확인한다(.).
- `seq 1888076871`: server의 초기 시퀀스 번호. 방향마다 따로 센다.
- `ack 2961618656`: client의 seq(...655) +1. "다음엔 656번부터 보내라". ACK는 늘 "다음에 기대하는 번호"다.
- 시각차 30마이크로초. 같은 호스트 내 가상 네트워크라 사실상 즉시.

### ③ ACK: "확인. 연결 성립" (client to server)

```text
15:05:16.598394  172.16.0.2.38552 > 172.16.0.1.9000:  Flags [.], ack 1, win 502, ...
```

- `Flags [.]`: 순수 ACK. 3단계. 여기서 3-way handshake 완성, 이제부터 데이터 전송 가능.
- `ack 1`: 이 지점부터 tcpdump가 상대 번호(relative)로 표시한다. 절대값 대신 "1 = 연결 시작점"으로 보여줘 읽기 쉽게 한다.
- 협상이 끝나 옵션은 더 붙지 않는다.

### ④ PSH-ACK: 실제 데이터 전송 (client to server)

```text
15:05:16.598516  172.16.0.2.38552 > 172.16.0.1.9000:  Flags [P.], seq 1:16, ack 1, length 15
```

- `Flags [P.]`: PSH + ACK. PSH는 "버퍼에 쌓지 말고 바로 앱에 올려라".
- `seq 1:16`: 1번부터 16번 직전까지 = 15바이트. `"hello-over-tcp\n"`가 정확히 15글자.
- `length 15`: seq 범위와 일치. 핵심은 연결을 다 세운 뒤에야 데이터가 실린다는 점.

### ⑤ ACK: "15바이트 잘 받았다" (server to client)

```text
15:05:16.598525  172.16.0.1.9000 > 172.16.0.2.38552:  Flags [.], ack 16, length 0
```

- `ack 16`: "1~15 다 받았고 다음은 16번부터". TCP의 신뢰성은 이 수신 확인에서 나온다. UDP엔 없다.

### ⑥⑦⑧ 연결 종료: FIN 교환

```text
15:05:17.601516  172.16.0.2 > 172.16.0.1:  Flags [F.], seq 16, ack 1   (client FIN: "나 끝")
15:05:17.601919  172.16.0.1 > 172.16.0.2:  Flags [F.], seq 1,  ack 17  (server FIN+ACK)
15:05:17.602038  172.16.0.2 > 172.16.0.1:  Flags [.],  ack 2           (client 최종 ACK)
```

- `Flags [F.]`: FIN + ACK. 연결을 정리(close)한다.
- 시각이 `15:05:17`로 데이터 전송보다 1초 뒤다. `nc -w1`의 1초 타임아웃이 끝나고 닫은 것.
- `ack 17`: 16바이트 데이터 + FIN 1칸 = 17번까지 처리했다는 뜻. FIN도 시퀀스 1칸을 차지한다.

정리하면 한 줄 메시지에 TCP는 패킷 8개를 썼다: 열기 3 + 데이터 1 + 확인 1 + 닫기 3.

## UDP: 단 한 줄

같은 포트로 `nc -u`를 한 번 쏜다.

```text
[5] [UDP] 같은 포트로 UDP 한 방 (핸드셰이크 없음)
      [client]$ echo "hello-over-udp" | nc -u -w1 172.16.0.1 9000
      server 수신 내용: hello-over-udp

      15:05:31.529118 IP 172.16.0.2.41732 > 172.16.0.1.9000: UDP, length 15
```

- Flags도 seq도 ack도, 핸드셰이크도 종료도 없다. 15바이트 한 방 던지고 끝.
- 출발 포트가 `41732`로 TCP(38552)와 다르다. 새 소켓이라 임시 포트도 새로 잡혔다.
- 시각 `15:05:31`은 TCP 구간(16~17초)과 14초 차이인데, 그사이 단계 진행 텀일 뿐 의미는 없다.
- 수신 확인(ACK)이 없으니 server가 받았는지 client는 알 수 없다. 빠르지만 신뢰성은 앱이 책임진다.

## 한눈 정리

| 항목 | TCP | UDP |
|---|---|---|
| 데이터 1건 전송에 쓴 패킷 | 8개 | 1개 |
| 연결 수립 | SYN, SYN/ACK, ACK | 없음 |
| 수신 확인(ACK) | 있음 (신뢰성) | 없음 |
| 종료 절차 | FIN 교환 | 없음 |
| 패킷에 붙는 정보 | seq / ack / win / flags / options | 길이뿐 |

TCP는 "연결을 세우고 한 바이트도 빠짐없이" 보내는 대신 오버헤드가 크고,
UDP는 "그냥 던지는" 대신 빠르고 가볍다. 같은 토폴로지에서 같은 15바이트를 보냈을 때
패킷 수가 8 대 1로 갈리는 것이 그 비용 차이를 그대로 보여준다.
