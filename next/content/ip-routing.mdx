## 라우팅: "다음 홉"의 연속

라우터는 전 세계 경로를 다 외우지 않습니다. 다만 **다음에 어디로 던질지**만 알고 있고,
그 다음 라우터가 또 다음 홉을 알고 있는 식으로 패킷이 흘러갑니다. 한 홉을 지날 때마다
TTL이 1씩 줄어들고, 0이 되면 폐기됩니다(루프 방지).

## NAT: 사설 IP를 공용 IP로 가리기

<Callout>
  <strong>NAT 뒤에서 외부 접근 받기</strong>는 본 페이지의 *연결 관점*과 다른 별개 운영 문제: reverse tunnel·STUN/TURN/ICE·bastion·ZTNA. 자세히는 <a href="/docs/advanced/nat-traversal">NAT Traversal</a> 페이지.
</Callout>


가정용 공유기는 한 개의 공인 IP 뒤에 사설 IP(`192.168.x.x`) 여러 대를 숨깁니다.
나갈 때 출발지 IP·포트를 공인 IP·새 포트로 바꾸고, 돌아오는 응답을 다시 사설 IP로 되돌리는 게 NAT(PAT)입니다.

<Callout>
  NAT는 IPv4 부족을 임시방편으로 해결했지만 부작용도 있습니다:
  외부에서 사설 IP로 직접 연결을 시작하기 어렵고(서버·P2P 곤란),
  애플리케이션이 자기 주소를 알아야 하는 프로토콜(SIP 등)은 별도 우회가 필요합니다.
</Callout>

## 라우터의 마음: Longest Prefix Match

라우팅 테이블에 여러 경로가 후보일 때, **가장 긴(=가장 구체적인) prefix가 이긴다**가 LPM 규칙입니다.

```
10.0.0.0/8    via R1
10.1.0.0/16   via R2
10.1.2.0/24   via R3
0.0.0.0/0     via Rdefault   ← 기본 경로
```

목적지 <code>10.1.2.42</code> → /24가 가장 구체적이므로 R3. 목적지 <code>10.1.5.1</code> → /16 매치 → R2. 목적지 <code>8.8.8.8</code> → 어디에도 안 맞아 기본 경로(/0) → Rdefault.

<Callout>
  하드웨어 라우터는 TCAM(Ternary CAM)으로 LPM을 한 클록에 처리.
  Linux 커널은 FIB의 trie 구조. 둘 다 결과는 같음: 가장 긴 매치.
</Callout>

## MTU와 Path MTU Discovery

링크마다 한 번에 보낼 수 있는 최대 크기가 있습니다. 이더넷은 1500바이트, PPPoE는 1492, IPsec/GRE 터널은 더 작음.

송신자가 1500바이트를 보냈는데 중간 링크가 1400밖에 못 받으면? 옛날엔 라우터가 패킷을 쪼개서 (IPv4 fragmentation) 보냈지만 성능·보안 이슈로 **DF(Don't Fragment) 비트**를 켜고 ICMP **"Fragmentation Needed"** 응답을 통해 송신자에게 알려주는 게 표준 (PMTUD).

<Callout tone="bad">
  <strong>"블랙홀"</strong>: 중간 방화벽이 ICMP를 죽이면 PMTUD 응답이 안 옴.
  TCP 핸드셰이크는 작은 패킷이라 성공하지만 실제 데이터 전송에서 멈춤: 디버깅 악몽.
  해법: <strong>MSS clamping</strong>: 라우터·LB가 SYN의 MSS 옵션을 강제로 작게 (예: 1380) 만들어 처음부터 작게 보내게 함.
</Callout>

## NAT 통과: STUN, TURN, ICE

두 호스트가 모두 NAT 뒤에 있으면 직접 연결 시도가 둘 다 막힙니다. 풀려면:

<KV>
  <KVItem term="STUN (Session Traversal Utilities for NAT)">
    호스트가 인터넷의 STUN 서버에 물어서 "내 공인 IP·포트는 뭐?"를 알아냄.
    그 정보를 시그널링 채널로 상대에게 전달. 둘이 동시에 외부로 패킷을 쏘면 NAT가 양쪽 매핑을 유지(hole punch).
  </KVItem>
  <KVItem term="TURN (Traversal Using Relays around NAT)">
    NAT 종류에 따라(symmetric NAT) hole punching 실패. 그땐 중계 서버를 통해 우회. 비용·지연 ↑.
  </KVItem>
  <KVItem term="ICE (Interactive Connectivity Establishment)">
    위 둘 + 직접 연결 후보를 모두 모아 동시 시도, 먼저 성공한 경로 채택. WebRTC가 사용.
  </KVItem>
</KV>

### NAT 분류 (구식 RFC 3489 vs 현행 RFC 4787)

<Callout tone="bad">
  <strong>RFC 3489 4분류(Full/Restricted/Port-Restricted/Symmetric Cone)는 2008년 deprecated.</strong>
  현행은 <strong>RFC 4787 behavior 분류</strong>: 
  <ul>
    <li><strong>Mapping behavior</strong>: Endpoint-Independent / Address-Dependent / Address-and-Port-Dependent</li>
    <li><strong>Filtering behavior</strong>: 위 셋과 독립적으로 또 나뉨</li>
  </ul>
  RFC 3489 용어가 STUN 문서·블로그에 아직 많이 남아 있어 둘 다 알아둘 것. 운영 핵심은 변함없음: Endpoint-Independent Mapping이면 hole punching 가능, Address-and-Port-Dependent (구 Symmetric)면 불가.
</Callout>

기억용 구식 4분류:
- **Full Cone** (현행: Endpoint-Independent Mapping + Filtering)
- **Restricted Cone** (현행: Endpoint-Independent Mapping + Address-Dependent Filtering)
- **Port Restricted** (현행: Endpoint-Independent Mapping + Address-and-Port-Dependent Filtering)
- **Symmetric** (현행: Address-and-Port-Dependent Mapping) → **hole punching 불가**, TURN 필요

## ECMP: 같은 비용, 여러 길

목적지로 가는 비용이 같은 경로가 여러 개면 **ECMP(Equal-Cost Multipath)** 로 분산합니다. 라우터는 5-tuple(src/dst IP, src/dst port, proto)을 해시해 결정: **같은 연결의 패킷은 항상 같은 경로**로.

<Callout tone="bad">
  <strong>ECMP 함정 1: 경로 변화 시 해시 깨짐</strong>. 경로가 추가/제거되면 해시 결과가 바뀌면서 진행 중인 연결의 패킷이 다른 경로로 가버림 → 순서 뒤집힘 또는 stateful 미들박스가 끊음. 해법: <strong>resilient hashing</strong>(consistent hash 변형)으로 변동 시에도 대부분 흐름이 같은 경로 유지.
</Callout>

<Callout tone="bad">
  <strong>ECMP 함정 2: 캡슐화에서 polarization</strong>. VXLAN/IPsec/GRE 같이 외부 헤더가 같은 5-tuple로 보이는 캡슐화에서는 내부 5-tuple이 가려져 ECMP가 한 경로로 쏠림(엔트로피 손실). 해법: <strong>entropy label (RFC 6790, MPLS)</strong> 또는 outer UDP src port에 inner 5-tuple 해시를 박아 인위적 엔트로피 주입 (VXLAN의 UDP src port가 정확히 이 용도). 운영자가 잘 모르면 멀티패스가 무용지물.
</Callout>

## IPv6 NDP: ARP의 후계자

IPv6에는 ARP가 없습니다. **NDP (Neighbor Discovery Protocol)** 가 ICMPv6로 같은 역할 + α를 수행:

<KV>
  <KVItem term="NS (Neighbor Solicitation)">ARP request 대응. "이 IPv6 주소의 MAC?"</KVItem>
  <KVItem term="NA (Neighbor Advertisement)">ARP reply 대응. 자기 MAC 응답.</KVItem>
  <KVItem term="RS / RA (Router Solicitation/Advertisement)">라우터가 자신의 존재와 prefix를 광고. 호스트는 RA를 받아 주소 자동 설정.</KVItem>
  <KVItem term="SLAAC (Stateless Address Autoconfiguration)">호스트가 RA의 prefix + 자기 인터페이스 ID로 IPv6 주소 자동 생성. DHCP 없이 IP 받음.</KVItem>
  <KVItem term="DAD (Duplicate Address Detection)">새 주소 사용 전에 같은 주소 있는지 확인.</KVItem>
</KV>

NDP도 인증이 없어 **NDP 스푸핑**이 가능. 방어: 실무 표준은 **RA Guard / DHCPv6 Guard / ND Inspection** 같은 *IPv6 First-Hop Security* 묶음 (스위치 단). SEND(Secure ND, RFC 3971)는 CGA 부담으로 거의 미배포: 이론서에선 자주 나오지만 실제는 안 씀.

## 라우팅 테이블 들여다보기

```bash
# Linux
ip route                     # 메인 테이블
ip route get 8.8.8.8         # 특정 목적지의 결정 경로
ip rule                      # 정책 라우팅 룰

# 다중 테이블 (정책 라우팅)
ip rule add from 10.0.1.0/24 table 100
ip route add default via 192.168.1.254 table 100

# IPv6 NDP 캐시
ip -6 neigh
```

<Callout>
  <strong>Policy routing</strong>의 흔한 사용: 두 ISP에 멀티홈한 서버에서
  "이 출발지 IP는 이쪽 ISP로" 같은 규칙. <code>ip rule</code>이 그 손잡이.
</Callout>
