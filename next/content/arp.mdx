## 스위치는 MAC으로만 전달한다

같은 스위치에 붙은 두 기기 A와 B가 있다고 합시다. A는 B의 IP는 알지만,
스위치는 이더넷 프레임을 MAC 주소로 전달합니다. 그래서 A는 먼저
"192.168.0.20을 가진 사람, 너의 MAC이 뭐야?"를 LAN 전체에 묻습니다.

## 합법 메커니즘인데 악용도 된다

**Gratuitous ARP (GARP)** 는 묻지도 않았는데 자기 IP↔MAC 매핑을 LAN에 *알리는* 패킷입니다. 그 자체로는 정상 동작:

- **IP 충돌 감지** (RFC 5227 ACD): 새 IP를 잡기 전 GARP로 "이 IP 이미 누가 쓰고 있나?" 확인
- **VRRP/keepalived failover**: VIP가 새 active 노드로 옮겨갈 때 GARP를 뿌려 스위치 MAC 테이블과 클라이언트 ARP 캐시 갱신
- **VM/컨테이너 마이그레이션**: 같은 이유로 새 위치에서 GARP

악용 시나리오: 공격자 X가 "192.168.0.1(게이트웨이)은 내 MAC이야"라고 같은 GARP 형식으로 *지속 송신* 하면, 피해자의 ARP 캐시가 덮어쓰이면서 외부로 나가는 모든 트래픽이 X를 거침. 자세한 시각화는 [보안 페이지](/docs/security)에서.

<Callout tone="bad">
  ARP는 "마지막에 들은 응답을 믿는" 단순한 모델입니다. 그래서 신뢰할 수 없는
  네트워크(공용 Wi-Fi 등)에서는 상위 계층 암호화(TLS)와 인증이 필수입니다.
</Callout>

<Callout>
  사내 모니터링이 GARP를 보고 무조건 알람 내면 keepalived failover마다 알람 폭주. <strong>GARP 자체는 정상</strong>, 비정상은 *동일 IP에 대한 GARP가 짧은 시간에 다른 MAC으로 반복* 같은 패턴.
</Callout>

<Eyebrow>운영: 캐시 상태</Eyebrow>

## ARP 캐시는 TTL이 있고 상태 머신이 있다

Linux의 ARP 캐시는 5가지 상태를 거칩니다:

<KV>
  <KVItem term="REACHABLE">최근 응답 확인됨. `base_reachable_time_ms` (기본 30s) 동안 유효.</KVItem>
  <KVItem term="STALE">유효 시간 지났지만 일단 사용. 다음 통신 시 확인.</KVItem>
  <KVItem term="DELAY">STALE 상태로 사용 후 짧은 시간 응답 대기.</KVItem>
  <KVItem term="PROBE">unicast ARP request로 재확인 중.</KVItem>
  <KVItem term="INCOMPLETE / FAILED">응답이 안 와서 해석 불가.</KVItem>
</KV>

VRRP/HA failover 후 일부 클라이언트가 한참 못 붙는 사고의 원인: **A의 ARP 캐시에 옛 MAC이 STALE/REACHABLE로 박혀 있어 GARP를 못 받았거나 무시**. 점검:

```bash
ip neigh                 # 캐시 + 상태 확인
ip neigh flush dev eth0  # 강제 비우기
arping -A -I eth0 10.0.0.1  # 직접 GARP 송신
```

## IPv6에는 ARP가 없다: NDP

IPv6는 **NDP (Neighbor Discovery Protocol)** 로 ARP 역할을 대신합니다. ICMPv6 type 133~137 위에 얹혀 동작 ([ICMP 페이지](/docs/icmp#icmpv6)).

**NDP 5 메시지**:

<DataTable>
  <thead>
    <tr><th>Type</th><th>이름</th><th>역할</th></tr>
  </thead>
  <tbody>
    <tr><td>133</td><td>Router Solicitation (RS)</td><td>"라우터 있어요?" 호스트가 부팅 시 송신</td></tr>
    <tr><td>134</td><td>Router Advertisement (RA)</td><td>라우터가 prefix · gateway · MTU · DNS 광고</td></tr>
    <tr><td>135</td><td>Neighbor Solicitation (NS)</td><td>ARP request 대체. "이 IP의 MAC은?"</td></tr>
    <tr><td>136</td><td>Neighbor Advertisement (NA)</td><td>ARP reply 대체. "그 IP는 내 것"</td></tr>
    <tr><td>137</td><td>Redirect</td><td>"더 좋은 경로 있어요"</td></tr>
  </tbody>
</DataTable>

### SLAAC: Stateless Address Autoconfiguration

IPv6는 *DHCP 없이* 자기 주소를 만들 수 있습니다.

<Steps>
  <li>호스트 부팅 시 RS 송신, 라우터가 RA 응답</li>
  <li>RA에 담긴 <strong>prefix</strong> (예: <code>2001:db8::/64</code>) + 자체 <strong>interface ID</strong> 64 bit 조합</li>
  <li>interface ID는 EUI-64 (MAC 기반) 또는 RFC 7217 stable privacy (해시 기반)</li>
  <li><strong>DAD (Duplicate Address Detection)</strong>: 만든 주소가 LAN에 이미 있는지 NS로 확인. 응답 없으면 사용 가능</li>
</Steps>

RA에는 prefix뿐 아니라 *주소 lifetime* (preferred · valid) 도 담깁니다. preferred 만료 시 새 통신은 다른 주소 우선.

### Privacy Extension (RFC 4941)

EUI-64는 *MAC을 그대로 노출*. 트래킹 위험. **Privacy Extension**: temporary IPv6 주소를 주기적으로 새로 생성. Windows · iOS · Android 기본 켜짐.

### DHCPv6 vs SLAAC

자동 구성 두 가지 선택 ([DHCP의 IPv6 섹션](/docs/dhcp#dhcpv6--slaac)):

<DataTable>
  <thead>
    <tr><th></th><th>SLAAC</th><th>Stateful DHCPv6</th><th>Stateless DHCPv6</th></tr>
  </thead>
  <tbody>
    <tr><td>주소</td><td>호스트 자체 생성</td><td>서버 할당</td><td>SLAAC</td></tr>
    <tr><td>DNS</td><td>RA의 RDNSS option 필요</td><td>옵션으로 제공</td><td>옵션으로 제공</td></tr>
    <tr><td>장점</td><td>가볍고 빠름</td><td>중앙 관리 · 추적</td><td>혼합 장점</td></tr>
    <tr><td>실무</td><td>홈 · 모바일 · 소규모</td><td>기업</td><td>대규모 LAN</td></tr>
  </tbody>
</DataTable>

NDP도 인증 없어 *NDP 스푸핑* 가능 → IPv6 First-Hop Security (RA Guard, ND Inspection) 필요. 자세한 보안 패턴은 [보안 페이지](/docs/security) 참조.
