<h2 id="arp-spoof">ARP 스푸핑</h2>

ARP는 응답을 검증하지 않습니다. 공격자 X가 "192.168.0.1(게이트웨이)은 내 MAC이다"라고
피해자 A의 ARP 캐시를 덮어쓰면, A가 외부로 보내는 모든 프레임이 X를 거쳐 갑니다.
X가 그대로 게이트웨이로 forwarding해 주면 A는 정상 동작처럼 느끼지만, 도청·변조가 가능해집니다.

### 방어

- 스위치에서 **DAI (Dynamic ARP Inspection)** + DHCP snooping
- 중요 호스트에 정적 ARP 항목
- 802.1X로 비인가 단말 자체를 LAN에 못 붙게
- L7 암호화(TLS): 경로는 빼앗겨도 내용은 못 읽게

<Eyebrow>02 · DNS</Eyebrow>

## DNS 캐시 포이즈닝

DNS 응답은 UDP·평문 기반입니다. 공격자가 리졸버보다 빨리 가짜 응답을 만들어
TX ID·소스 포트를 맞추면, 리졸버는 그것을 진짜로 믿고 TTL 동안 캐시합니다.
그 캐시를 보는 모든 사용자가 가짜 IP로 안내됩니다.

<Callout tone="bad">
  <strong>Kaminsky 공격의 본질 (2008)</strong>. DNS TX ID는 **16-bit = 65,536**. 공격자가 64K 위조 응답을 *동시에* 던지면 확률적으로 맞춤. 더 무서운 점은 **bailiwick rule 악용**: `random123.bank.com`을 묻고 위조 응답에 `bank.com NS = attacker.com` glue까지 끼워넣어 *bank.com 전체 권한을 한 번에 탈취*. 캐시 한 번 오염되면 TTL 내내 가짜 IP. **소스 포트 무작위화(16-bit 추가)** 가 표준 대응 → 합쳐서 32-bit = 약 40억, 확률적 공격 불가능.
</Callout>

### 방어

- **소스 포트 무작위화**(Kaminsky 사건 후 표준화): 추측 공간을 16-bit 추가 = 32-bit로 확장
- **0x20 인코딩** (RFC 5452 보강): 질의 이름의 대소문자를 무작위화(`gOoGLe.cOm`), 응답이 같은 대소문자로 와야 수락. 추가 엔트로피.
- **DNS Cookie** (RFC 7873): 클라이언트·서버가 짧은 cookie 교환으로 약식 인증. DNSSEC 미배포 환경의 현실 방어.
- **DNSSEC**: 응답의 디지털 서명을 검증 (배포율 낮음, KSK rollover 운영 비용)
- **DoT/DoH**: 도청·삽입 공격 자체를 차단

## SYN Flood: 자원 고갈

TCP 서버는 SYN을 받으면 "half-open" 상태로 자원을 잡고 SYN+ACK를 보냅니다.
공격자가 출발지 IP를 위조해 수많은 SYN만 던지면, 서버의 백로그 큐가 가득 차서
정상 사용자의 연결을 받지 못합니다.

### 방어

- **SYN cookie**: 백로그를 잡지 않고 응답 안에 상태를 인코딩
- 방화벽·로드밸런서의 SYN proxy / rate limit
- uRPF, 소스 IP 검증으로 위조 트래픽 일부 차단

<Callout tone="bad">
  <strong>SYN cookie의 트레이드오프</strong>. 켜면 백로그 큐 부담은 사라지지만 TCP 옵션(window scale, SACK, timestamp)이 인코딩 공간 부족으로 손실됨 → 정상 사용자 성능 저하. 그래서 Linux는 평시 OFF, **백로그 임계 초과 시에만 자동 ON** (<code>net.ipv4.tcp_syncookies=1</code>). 모니터링: `netstat -s | grep -i listen` 백로그 드롭 카운터.
</Callout>

<Eyebrow>04 · UDP</Eyebrow>

## 반사·증폭 DDoS

공격자가 작은 요청에 큰 응답을 돌려주는 서비스(DNS, NTP, memcached)에
피해자의 IP를 출발지로 위조해 요청을 던집니다. 응답은 피해자에게 쏟아져,
공격자는 자기 대역폭의 수십~수만 배를 폭격에 동원할 수 있습니다.

<DataTable>
  <thead><tr><th>서비스</th><th>BAF (Bandwidth Amplification Factor)</th><th>이유</th></tr></thead>
  <tbody>
    <tr><td>DNS (ANY query)</td><td>28~54x</td><td>큰 레코드(특히 DNSSEC 서명·키)</td></tr>
    <tr><td>NTP (monlist)</td><td>~556x</td><td>최근 클라이언트 목록 반환</td></tr>
    <tr><td>memcached (인증 없음)</td><td>~51,000x</td><td>대량 키 응답: 2018 GitHub 1.35Tbps 공격의 정체</td></tr>
    <tr><td>SSDP / SNMPv2</td><td>~30x / ~6x</td><td>IoT 기기·라우터에 노출된 UPnP·SNMP</td></tr>
  </tbody>
</DataTable>

<Callout tone="bad">
  실무에서 BAF가 높은 서비스를 인터넷에 노출하는 게 가장 흔한 사고 원인. memcached UDP 11211을 인터넷에 열어두면 며칠 안에 봇이 발견해 51,000배로 증폭 → 1 Gbps 봇이 51 Tbps 트래픽 생성. 점검: `nmap -sU -p 11211 your-edge-ip`.
</Callout>

### 방어

- ISP의 **BCP38**(출발지 위조 차단): 가장 근본적
- 증폭 가능한 서비스를 인터넷에 노출하지 않음
- 응답 크기 제한, 인증 강제
- 방어 측: CDN/anycast로 트래픽 분산, 스크러빙 센터

## DDoS 방어의 layer 분담

같은 "DDoS 막기"가 *어느 layer에서 무엇을* 하는지 명확히 분담해야 운영이 됩니다.

<DataTable>
  <thead><tr><th>Layer</th><th>공격 유형</th><th>방어 위치</th><th>도구·기법</th></tr></thead>
  <tbody>
    <tr><td><strong>L3/L4 volumetric</strong></td><td>UDP flood, ICMP flood, 반사·증폭</td><td>인터넷 라우팅 layer + 클라우드 엣지</td><td>BGP FlowSpec, RTBH, Anycast scrubbing, AWS Shield, Cloudflare Magic Transit</td></tr>
    <tr><td><strong>L4 stateful</strong></td><td>SYN flood, ACK flood, connection exhaustion</td><td>edge LB / 클라우드 게이트웨이</td><td>SYN cookie, connection rate limit, conntrack tuning</td></tr>
    <tr><td><strong>L7 application</strong></td><td>HTTP flood, slowloris, cache busting, credential stuffing</td><td>CDN edge / WAF / API gateway</td><td>WAF rules, JS challenge·CAPTCHA, rate limit, bot detection</td></tr>
    <tr><td><strong>L7 business logic</strong></td><td>API abuse, scraping, fraudulent transaction</td><td>application</td><td>per-user rate limit, behavioral analysis, MFA step-up</td></tr>
  </tbody>
</DataTable>

<Callout>
  **운영 원칙**: 위 layer는 *순차*가 아니라 *동시*. 각 layer가 *자기 책임*만 진다. L3/L4를 application에서 막으려 들거나 L7을 라우팅 layer에서 막으려 들면 비효율적.
</Callout>

자세한 도구·운영은 인접 페이지로 분담:
