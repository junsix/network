## "Default deny inbound + default allow outbound"가 표준

업계 관습:
- **Inbound는 기본 차단**: 명시적으로 허용한 포트·소스만 통과.
- **Outbound는 기본 허용**: 서버가 무엇이든 외부로 나갈 수 있음.

### 왜 이렇게 비대칭인가

<PCBox>
  <PC tone="good" title="Inbound 엄격의 이유">
    <ul>
      <li>알려진 서비스만 노출해 공격 표면 최소화</li>
      <li>인증 없는 외부 접근을 1차 차단</li>
      <li>SG·NACL·방화벽 룰의 핵심 책무</li>
    </ul>
  </PC>
  <PC tone="bad" title="Outbound 관대의 부작용">
    <ul>
      <li>침해된 서버가 C2(command-and-control)로 외부 신호 발사</li>
      <li>크리덴셜·데이터 유출 (DNS·HTTPS로 위장)</li>
      <li>크립토 마이너·스팸 봇이 자유롭게 통신</li>
    </ul>
  </PC>
</PCBox>

<Callout tone="bad">
  <strong>Egress filtering</strong>은 보안의 마지막 보루로 점점 표준이 되는 중.
  "내부 서버는 인터넷의 정해진 도메인 외에는 outbound 금지": Zero Trust의 핵심 실천.
  단, 운영 비용·디버깅 난이도가 inbound 정책보다 훨씬 큼.
</Callout>

<Eyebrow>호스트 방화벽</Eyebrow>

## Linux iptables / nftables의 3 체인

```
        ┌─────────┐         ┌─────────┐         ┌──────────┐
입력 →  │  INPUT  │ →  로컬 │ FORWARD │ →  출력 │  OUTPUT  │  → 외부
        └─────────┘  프로세스 └─────────┘         └──────────┘
        (이 호스트가  (이 호스트를     (이 호스트가
         받는 패킷)    경유하는 패킷)   보내는 패킷)
```

<KV>
  <KVItem term="INPUT">이 호스트로 들어와 *로컬 프로세스에 전달*되는 패킷. 서버 방화벽의 inbound 규칙이 여기.</KVItem>
  <KVItem term="OUTPUT">로컬 프로세스가 *밖으로 보내는* 패킷. egress 규칙.</KVItem>
  <KVItem term="FORWARD">이 호스트를 경유하지만 종착지가 아닌 패킷. 라우터·NAT·K8s 노드에서 사용.</KVItem>
  <KVItem term="PREROUTING">라우팅 결정 직전. DNAT가 여기서 일어남 (LB의 VIP → 백엔드 IP 변환).</KVItem>
  <KVItem term="POSTROUTING">라우팅 후 송출 직전. SNAT가 여기 (사설 IP → 공인 IP).</KVItem>
</KV>

### 기본 명령

```bash
# 모든 inbound 차단 (default deny inbound)
iptables -P INPUT DROP

# loopback 허용 (필수)
iptables -A INPUT -i lo -j ACCEPT

# established 연결의 응답 패킷은 허용 (stateful)
iptables -A INPUT -m conntrack --ctstate ESTABLISHED,RELATED -j ACCEPT

# SSH·HTTP·HTTPS만 inbound 허용
iptables -A INPUT -p tcp --dport 22  -j ACCEPT
iptables -A INPUT -p tcp --dport 80  -j ACCEPT
iptables -A INPUT -p tcp --dport 443 -j ACCEPT

# outbound는 모두 허용 (default)
iptables -P OUTPUT ACCEPT
```

<Callout>
  <strong>nftables</strong>는 iptables의 후속 (Linux 3.13+). 문법이 더 일관되고 IPv4/v6 통합.
  운영은 점진적으로 nftables로 이동 중이지만 K8s 노드에선 여전히 iptables가 다수.
</Callout>

[클라우드 네트워킹 페이지](/docs/advanced/cloud-net)와 겹치지만, 방향성 관점에서 다시 정리:

  </tbody>
</DataTable>

### "Stateful"이 정확히 무슨 의미?

면접 단골 질문. SG는 conntrack 비유로 동작합니다:

<Steps>
  <li>SG inbound에 <code>22/tcp from 0.0.0.0/0</code> 허용 룰만 있고 outbound는 없음 (전부 차단)</li>
  <li>외부 SSH 클라이언트 → 서버: SYN → SG inbound 룰 매치, 통과</li>
  <li>서버 → 외부: SYN+ACK → outbound 룰 *없는데도* 자동 허용 (stateful)</li>
  <li>이후 같은 연결의 모든 패킷이 conntrack 추적으로 양방향 통과</li>
</Steps>

NACL은 stateless이므로 같은 시나리오에서 outbound 룰에 ephemeral 포트(32768-60999) 허용을 명시해야 응답이 나감. **NACL이 까다로운 이유**.

<Callout tone="bad">
  NACL을 "deny 1.2.3.4"용으로만 좁게 쓰고, 메인 정책은 SG에 두는 게 운영 표준.
  NACL에 잘못된 ephemeral port deny를 넣으면 outbound 응답이 안 나가 디버깅 지옥.
</Callout>

## ingress + egress 양쪽 설계

K8s NetworkPolicy는 두 방향을 *별도로* 정의합니다:

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: api-policy
spec:
  podSelector:
    matchLabels: { app: api }
  policyTypes: [Ingress, Egress]      # ← 둘 다 명시
  ingress:
    - from:
        - podSelector:
            matchLabels: { app: frontend }
      ports:
        - { protocol: TCP, port: 8080 }
  egress:
    - to:
        - podSelector:
            matchLabels: { app: db }
      ports:
        - { protocol: TCP, port: 5432 }
    - to:                              # DNS 허용 (잊으면 도메인 조회 못 함)
        - namespaceSelector:
            matchLabels: { name: kube-system }
          podSelector:
            matchLabels: { k8s-app: kube-dns }
      ports:
        - { protocol: UDP, port: 53 }
```

### 자주 빠뜨리는 egress 룰

<DataTable>
  <thead><tr><th>잊으면 안 되는 egress</th><th>증상</th></tr></thead>
  <tbody>
    <tr><td>kube-dns(CoreDNS) UDP 53</td><td>DNS 조회 실패 → 모든 호출이 호스트네임 해석 안 됨</td></tr>
    <tr><td>kube-apiserver 443</td><td>ServiceAccount 토큰 검증 실패, 일부 SDK가 멈춤</td></tr>

<Callout>

  관측을 충분히 한 뒤 egress도 단계별로 좁힘. 한 번에 양쪽 다 잠그면 운영 사고 확률 ↑.

## SNAT vs DNAT: 누가 어느 방향에서 변환되나

<KV>
  <KVItem term="SNAT (Source NAT)">
    <strong>나가는 패킷의 src IP를 변경</strong>. 사설 IP(192.168.x.x) → 공인 IP.
    가정용 공유기·NAT Gateway가 하는 일. <strong>outbound 흐름</strong>의 시작.
  </KVItem>
  <KVItem term="DNAT (Destination NAT)">
    <strong>들어오는 패킷의 dst IP를 변경</strong>. VIP(서비스 IP) → 백엔드 파드 IP.
    K8s ClusterIP·LB·포트포워딩의 본질. <strong>inbound 흐름</strong>의 시작.
  </KVItem>
  <KVItem term="MASQUERADE">
    SNAT의 한 종류. src IP를 송출 인터페이스의 IP로 자동 변환. 
    IP가 DHCP로 자주 바뀌는 환경에 적합.
  </KVItem>
  <KVItem term="Hairpin NAT">
    내부 클라이언트가 공인 IP를 통해 같은 내부 서버로 접속할 때.
    SNAT + DNAT가 동시에 일어남. 가정용 공유기에서 자주 안 됨.
  </KVItem>
</KV>

<Eyebrow>로드밸런서</Eyebrow>

## 한 LB에서 inbound/outbound가 동시에

```
  [Client]  ──inbound──►  [LB]  ──outbound──►  [Backend]
            (LB 관점)            (LB 관점)
                                 ◄──inbound──  (Backend 관점)
            ◄──outbound──         ◄──outbound── (응답)
```

같은 연결 안에서 LB는 4번의 방향성 변환을 처리:

<Steps>
  <li>Client → LB: LB의 **inbound** (외부 listener에 도착)</li>
  <li>LB → Backend: LB의 **outbound** (내부 백엔드로 전달, source IP는 LB 또는 보존)</li>
  <li>Backend → LB: LB의 **inbound** (응답 도착)</li>
  <li>LB → Client: LB의 **outbound** (클라이언트로 응답)</li>
</Steps>

<Callout>
  L4 LB는 보통 <strong>DSR (Direct Server Return)</strong>로 outbound를 최적화: 응답을 LB 거치지 않고 backend가 직접 client에게.
  대역폭 절감 효과 큼. 단, SG·NACL 룰이 비대칭 흐름에 맞게 설계돼야.
</Callout>

<Eyebrow>conntrack</Eyebrow>

## stateful의 비밀

iptables·SG·서비스 메시 모두 **conntrack** 테이블에 의존합니다. 흐름 단위 상태 머신:

```
[NEW] → [ESTABLISHED] → [RELATED] (응답·ICMP 에러)
       ↓
       → [INVALID] (룰에 안 맞는 패킷)
       → [CLOSED]
```

```bash
# 현재 추적 중인 연결
conntrack -L

# TCP 연결 한정
conntrack -L -p tcp

# 특정 IP 관련만
conntrack -L -d 10.0.1.5
```

conntrack 테이블이 가득 차면 새 연결이 거부됨. NAT 노드·LB·k8s 워커에서 자주 부딪치는 운영 이슈:

```bash
# 현재 사용량
cat /proc/sys/net/netfilter/nf_conntrack_count
cat /proc/sys/net/netfilter/nf_conntrack_max

# 늘리기
sysctl -w net.netfilter.nf_conntrack_max=1048576
```

## AS 사이의 inbound·outbound 정책

[BGP 페이지](/docs/advanced/bgp)에서 다룬 LOCAL_PREF/AS_PATH prepend는 사실 *방향성 정책*입니다:

<DataTable>
  <thead><tr><th>제어 대상</th><th>손잡이</th><th>방식</th></tr></thead>
  <tbody>
    <tr><td>나가는 트래픽 (outbound)</td><td>LOCAL_PREF</td><td>내부 정책. 어느 이웃을 통해 나갈지 우리가 정함.</td></tr>
    <tr><td>들어오는 트래픽 (inbound)</td><td>AS_PATH prepend, MED</td><td>외부 광고에 영향. "이쪽 경로는 길어 보이게" 만들어 다른 경로 유인.</td></tr>
  </tbody>
</DataTable>

<Callout>
  <strong>대칭 라우팅 보장은 어렵다</strong>: 우리가 LOCAL_PREF로 outbound를 통제해도 들어오는 트래픽 경로는 상대방 AS의 정책이라 통제 불가.
  비대칭 라우팅이 자주 발생: stateful 방화벽이 한쪽만 보고 끊는 사고로 이어짐.
</Callout>

## 방향성 관련 자주 묻는 질문

- "SG가 stateful이라는 게 정확히 무슨 의미?" → outbound 룰 없이도 inbound 흐름의 응답이 자동 허용. conntrack 추적.
- "NACL은 왜 양방향 룰을 따로 써야 하나?" → stateless라서. ephemeral 포트 outbound 룰 빠뜨리면 응답이 안 나감.
- "Default deny inbound + allow outbound가 표준인 이유?" → 인바운드는 공격 표면, 아웃바운드는 정상 활동의 95%. 그러나 침해 후 C2 차단을 위해 egress filtering 점차 표준화.
- "K8s NetworkPolicy에서 egress 룰을 잠갔는데 파드가 동작 안 한다." → CoreDNS (UDP 53), kube-apiserver(443) egress 누락 의심.
- "DSR이 왜 outbound 트래픽 절감인가?" → 응답 패킷이 LB를 안 거치고 backend → client 직접. LB 대역폭 큰 폭 절약.
- "비대칭 라우팅이 stateful 방화벽에서 왜 문제?" → SYN은 경로 A로 갔는데 SYN+ACK가 경로 B로 돌아오면 B의 방화벽은 그 흐름의 "NEW"를 못 봐서 INVALID로 drop.
- "egress 비용이 inbound보다 비싼 이유?" → 클라우드 사업자가 사용자 lock-in을 만드는 가격 정책. 인터넷은 본질적으로 양방향이지만 사업적 비대칭.
- "BGP에서 들어오는 트래픽을 통제하기 어려운 이유?" → 들어오는 경로는 상대 AS의 정책이라 우리가 강제 불가. AS_PATH prepend·MED로 *유인*만 가능.
