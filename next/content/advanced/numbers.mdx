## 커넥션·소켓·파일 디스크립터 한계

<KV>
  <KVItem term="65,535 ports">단일 IP에서 동시 외부 연결의 절대 한계. SO_REUSEPORT로 늘리거나 source IP 추가.</KVItem>
  <KVItem term="Linux 기본 fd limit">소프트 1024, 하드 65536. <code>ulimit -n</code>·systemd LimitNOFILE 조정 필요.</KVItem>
  <KVItem term="conntrack 테이블">기본 ~262K. Kubernetes 노드에선 자주 부족 → <code>net.netfilter.nf_conntrack_max</code> 증가.</KVItem>
  <KVItem term="TCP 메모리">각 idle 연결 ~4KB. 100만 동시 연결 = ~4GB. C10K 시대 이후 C1M은 표준.</KVItem>
  <KVItem term="ephemeral port range">Linux 기본 <code>32768-60999</code> = 약 28K. 짧은 연결 많으면 부족.</KVItem>
</KV>

<KV>
  <KVItem term="CPU > 80% 지속 5분">스케일 아웃 신호. 단, 노이즈(데이터플레인의 정상 활동) 구분 필요.</KVItem>

## 숫자가 답이 되는 질문들

- "초당 100만 요청을 받는 시스템 설계?" → 단일 머신 처리량 → 필요 머신 수 → 네트워크 PPS → DB 샤딩.
- "한국 사용자에게 미국 origin은 너무 느립니다." → RTT 180ms 기준: TLS 1.3이면 TCP 1 RTT + TLS 1 RTT = 2 RTT ≈ 360ms (TLS 1.2면 3 RTT ≈ 540ms). CDN 엣지에서 끝나면 5ms.
- "월 100TB egress 비용?" → 100TB × $0.09/GB ≈ $9,200 (10TB 이하 구간 무시). CDN로 캐시 hit 90% 가정 시 10TB만 origin egress.
- "100만 동시 연결 서버는 메모리 얼마?" → idle 연결당 커널 socket 구조체 + 기본 rmem/wmem 합쳐 **~20~50KB** (4KB는 Go goroutine 스택 단독 수치). 100만 연결 ≈ 20~50GB. 짧은 연결·HTTP/2 멀티플렉싱·tuning(rmem 작게)으로 줄일 수 있지만 "4KB×100만=4GB"는 너무 낙관적.
- "DB가 5000 tps 한계인데 10000 tps 필요?" → 읽기 캐시·write batching·샤딩·CQRS.
- "AZ 간 통신 비용이 월 1000달러 넘는다?" → 동일 AZ 배치(topology-aware) + 캐시. 1000달러 = 50TB 트래픽.
- "SLO 99.9%면 월 다운타임 예산?" → 43분. 한 번 사고로 30분 다운 = 예산의 70% 소진.
