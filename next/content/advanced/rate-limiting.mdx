<Callout>
  Cloudflare·Stripe API가 **GCRA**로 옮긴 이유: token bucket과 동등한 의미를 *상태 1 timestamp*로 표현 가능 → Redis 단일 키로 분산 운영이 가벼움.
</Callout>

<Callout>
  <strong>GCRA가 token bucket과 어떻게 같은지 (직관)</strong>. "버킷에 N개 토큰" 상태를 *"다음 요청이 허용되는 가상 시각 TAT"* 로 변환한 것. 토큰 가득 차면 TAT = 현재 시각(즉시 통과). 1개 소비할 때마다 TAT가 <code>period</code>만큼 미래로 이동. <code>burst_tolerance</code>만큼 미래로 가도 허용. 즉 *상태가 timestamp 1개*. 토큰 카운터 대신 시계 한 점만 추적: 분산에서 한 키만 atomic 갱신하면 됨.
</Callout>

## Token Bucket과 GCRA

```python
# Token Bucket (전통)
def allow(key, capacity=10, refill_rate=5):
    now = time.time()
    state = redis.hgetall(key)
    tokens = float(state.get("tokens", capacity))
    last = float(state.get("ts", now))
    # 경과 시간만큼 토큰 충전
    tokens = min(capacity, tokens + (now - last) * refill_rate)
    if tokens >= 1:
        tokens -= 1
        redis.hset(key, {"tokens": tokens, "ts": now})
        return True
    redis.hset(key, {"tokens": tokens, "ts": now})
    return False
```

```python
# GCRA: TAT 한 값으로
def allow_gcra(key, period=0.2, burst_tolerance=2.0):
    # period = 1 / rate, burst_tolerance = 허용 가능 burst (초)
    now = time.time()
    tat = float(redis.get(key) or 0)
    new_tat = max(tat, now) + period
    if new_tat - now <= burst_tolerance:
        # TTL은 max(period, burst_tolerance) × 2: period > burst_tolerance면
        # int(burst_tolerance)+1로는 다음 정당한 요청 전 TAT 만료 → burst 우회.
        ttl = int(max(period, burst_tolerance) * 2) + 1
        redis.set(key, new_tat, ex=ttl)
        return True
    # Retry-After는 *델타*(초). 절대 시각 아님 (RFC 9110).
    retry_after = max(0, (new_tat - burst_tolerance) - now)
    return False  # 응답 헤더: Retry-After: {int(retry_after)}
```

<Callout tone="bad">
  위 의사 코드는 *원자성*이 없음 (get·계산·set 사이 race). 운영에선 반드시 Redis **Lua 스크립트**(Redis는 *single-threaded*라 한 스크립트 안 모든 명령이 *원자적 실행* 보장) 또는 <code>SET ... NX</code>로 감싸야 함. Stripe·GitHub의 공개 구현 참조.
</Callout>

<Eyebrow>Thundering Herd</Eyebrow>

## 동시 재시도 폭풍 회피

거절당한 N개 클라이언트가 정확히 같은 시각에 재시도하면 *bursting 거절*이 영원히 반복 (oscillation).

<Steps>
  <li><strong>Jittered backoff</strong>: <code>delay = base × 2^attempt × random(0.5, 1.5)</code></li>
  <li><strong>AWS full jitter</strong>: <code>delay = random(0, base × 2^attempt)</code> : 동시성 가장 효과적으로 분산</li>
  <li><strong>Decorrelated jitter</strong>: <code>delay = random(base, prev × 3)</code> : 진동 줄임</li>
  <li><strong>Request coalescing</strong>: 같은 요청은 *하나만* backend로, 응답을 N에 broadcast</li>
  <li><strong>Stampede prevention</strong>: 캐시 만료 시 *하나의 키*만 갱신 (probabilistic early expiration)</li>
</Steps>

<Callout tone="bad">
  Redis 캐시 만료 직후 N개 요청이 동시에 origin 직격하는 사례 : *probabilistic early refresh* (e.g., XFetch) 또는 *lock-on-miss*로 단일화. 자세한 패턴은 <a href="/docs/advanced/reliability">신뢰성 패턴</a> 페이지의 stampede 섹션.
</Callout>

## Fail-open vs Fail-closed

Rate limit의 가장 큰 운영 결정: *Redis가 죽으면 모든 요청을 통과시킬 것인가(fail-open) 거절할 것인가(fail-closed)?*

<DataTable>
  <thead><tr><th></th><th>Fail-open</th><th>Fail-closed</th></tr></thead>
  <tbody>
    <tr><td>Redis 장애 시</td><td>cap 없이 통과 (가용성 우선)</td><td>모두 거절 (보호 우선)</td></tr>
    <tr><td>적합</td><td>가용성 SLO가 중요한 일반 API</td><td>보안 게이트웨이·인증·결제</td></tr>
    <tr><td>대표</td><td>Stripe API</td><td>다수 보안 게이트웨이</td></tr>
    <tr><td>도구 토글</td><td>envoyproxy/ratelimit <code>denyOnFail: false</code></td><td><code>denyOnFail: true</code></td></tr>

<Callout tone="bad">
  시니어 면접 단골 질문. "Redis 죽으면 어떻게 됩니까?"에 답이 없으면 *즉시 감점*. 정답은 *결정의 근거*까지: "가용성 SLO가 높은 일반 API는 fail-open, 결제·인증은 fail-closed"식으로 *책임에 맞춰* 선택. 모니터링도 둘이 분리 (fail-open은 *통과 비율 알람*, fail-closed는 *거절 비율 알람*).
</Callout>

<Eyebrow>In-flight 제한</Eyebrow>

## RPS vs Concurrency: Little's Law

RPS만 제한하면 응답이 느려질 때 backend의 in-flight 요청 수가 *자동으로 차오른다*. 결국 메모리·DB 커넥션 풀이 먼저 깨진다.

**Little's Law**: <strong>L = λ × W</strong> (in-flight = RPS × 응답 시간). 응답 W가 두 배 느려지면 in-flight L도 두 배.

<KV>
  <KVItem term="동시성 제한">in-flight cap 설정. 그 이상은 wait 또는 즉시 거절. backpressure가 자연스럽게 발생.</KVItem>
  <KVItem term="Netflix concurrency-limits">Vegas-style adaptive limit. latency를 보고 동적 조정 (TCP 혼잡 제어 알고리즘 차용).</KVItem>
  <KVItem term="Envoy adaptive_concurrency_filter">latency·error rate 기반 동적 cap. 정적 임계 튜닝 불필요.</KVItem>
  <KVItem term="실무">두 단계 cascade: 거친 RPS cap (DDoS 1차) + 정밀 concurrency cap (backend 보호)</KVItem>
</KV>

<Eyebrow>적응형</Eyebrow>

## Adaptive Rate Limiting (AIMD · graceful degradation)

정적 cap은 트래픽 패턴이 변하면 따라가지 못한다. **AIMD** (Additive Increase Multiplicative Decrease, TCP 혼잡 제어와 같은 원리)를 차용해 cap을 동적으로 조정.

<Steps>
  <li>정상이면 cap을 *선형 증가* (예: 매 분 +1%)</li>
  <li>backpressure 신호(latency 임계 초과, error rate, queue depth) 감지 시 cap을 *곱셈 감소* (×0.5)</li>
  <li>Google "graceful degradation"·Netflix Vegas-style·Envoy adaptive concurrency가 이 패턴</li>
</Steps>

<DataTable>
  <thead><tr><th>전략</th><th>정확도</th><th>지연</th></tr></thead>
  <tbody>
    <tr><td>중앙 region에 카운터</td><td>완전 정확</td><td>cross-region RTT (수십~수백ms)</td></tr>
    <tr><td>region별 sub-budget (예: 글로벌 1000 → us-east 600 / eu 400)</td><td>전체는 글로벌 cap 이하 보장</td><td>지역 RTT만</td></tr>
    <tr><td>CRDT G-Counter로 region별 합산 (gossip)</td><td>약간 over (CRDT eventual)</td><td>지역 RTT</td></tr>
    <tr><td>best-effort 글로벌 통계 + 거친 region cap</td><td>근사</td><td>지역 RTT</td></tr>
  </tbody>
</DataTable>

<Callout>
  Stripe·Cloudflare 같은 글로벌 서비스는 *sub-budget* + *gossip 합산*의 hybrid. 완전 정확은 *너무 비싸서* 약간 over하더라도 가용성을 선택.
</Callout>

## AWS jitter 정확 분류 + Redis 함정

AWS Architecture 블로그(Marc Brooker, *Exponential Backoff And Jitter*)의 정식 분류:

<DataTable>
  <thead><tr><th>jitter 종류</th><th>공식</th><th>특징</th></tr></thead>
  <tbody>
    <tr><td>No jitter</td><td><code>delay = base × 2^attempt</code></td><td>동시 클라이언트 그대로 동기화: 폭주</td></tr>
    <tr><td>Equal jitter</td><td><code>temp/2 + random(0, temp/2)</code> where <code>temp = base × 2^attempt</code></td><td>약한 분산</td></tr>
    <tr><td><strong>Full jitter</strong></td><td><code>random(0, base × 2^attempt)</code></td><td>가장 효과적으로 동시성 분산. 대규모 클라이언트 권장</td></tr>
    <tr><td><strong>Decorrelated jitter</strong></td><td><code>min(cap, random(base, prev × 3))</code></td><td>진동 줄임. <strong>cap 필수</strong> (없으면 무한 증가)</td></tr>
  </tbody>
</DataTable>

<Callout tone="bad">
  Redis <code>maxmemory-policy=allkeys-lru</code>이면 rate limit 키가 *eviction*되어 burst 우회 가능. <code>volatile-lru</code> + 명시적 TTL + dedicated rate-limit Redis 인스턴스가 정석.
</Callout>
