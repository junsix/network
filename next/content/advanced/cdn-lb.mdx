## 왜 거리가 문제인가

빛은 진공에서 300,000 km/s. 한국–미국 동부 약 11,000km, 편도 = 거리 ÷ (c/n) × 굴절률 1.5 → 55ms, RTT ≈ 110ms.
여기에 큐잉·라우팅·TLS handshake가 더해지면 250ms는 우습습니다.
TCP 3-way handshake는 **1 RTT** (SYN → SYN+ACK → ACK 이후 첫 데이터), TLS 1.3은 추가 **1 RTT** (0-RTT면 0). 합쳐서 **TCP + TLS 1.3 = 2 RTT ≈ 220ms** 가 한 번 손실되고 시작합니다.

## 어떤 서버로 보낼지 정하는 법

<KV>
  <KVItem term="round-robin">순서대로 돌아가며 보낸다. 가장 단순, 가중치 옵션 있음. 서버 부하 차이는 못 봄.</KVItem>
  <KVItem term="least-connections">현재 연결이 가장 적은 서버로. 처리 시간이 다양한 워크로드에 강함.</KVItem>
  <KVItem term="least-response-time">최근 응답이 빠른 쪽 선호. 느려진 서버를 자동 배제하는 효과.</KVItem>
  <KVItem term="consistent hashing">요청의 키(예: 사용자 ID)를 해시해 항상 같은 서버로. <strong>서버 수가 바뀌어도 대부분의 키는 같은 서버에 머묾</strong>: 캐시 적중률 보존.</KVItem>
  <KVItem term="maglev / rendezvous">consistent hash의 진화형. Google Maglev은 충돌 해소 테이블을 미리 만들어 두고 lookup이 O(1).</KVItem>
  <KVItem term="EWMA">지수 이동 평균으로 서버별 응답시간을 추적하며 더 빠른 곳에 가중치.</KVItem>
  <KVItem term="P2C (Power of Two Choices)">무작위 두 서버 중 부하 적은 쪽 선택. 거의 균등 분산을 매우 저렴하게 달성.</KVItem>
  <KVItem term="health check">주기적인 핑/HTTP probe로 비정상 서버 제외. fail-open vs fail-close 정책 선택.</KVItem>
</KV>

## sticky session: 같은 사용자를 같은 서버로

세션 상태가 서버 로컬에 있으면, 다음 요청도 같은 서버로 보내야 합니다.
쿠키 주입(LB가 세션 ID를 쿠키로 발급) 또는 클라이언트 IP 해시로 구현합니다.
(consistent hashing은 *분산 LB 알고리즘*: 노드 추가/제거 시 재해싱 최소화 목적이라 sticky session 구현과는 다른 개념. 캐시 키 라우팅에 쓰임.)
결국 서버가 죽으면 그 세션도 잃습니다. 가능하면 세션을 외부 저장소(Redis 등)에 두는 편이 낫습니다.

## Anycast Scrubbing · Edge Challenge

Anycast의 가장 큰 *전략적* 가치는 부하 분산 자체가 아니라 **DDoS 흡수**입니다. 같은 IP를 전 세계 수백 edge가 광고하므로 단일 지역의 폭주를 *수학적으로* 분산.

<DataTable>
  <thead><tr><th>방어 위치</th><th>기법</th><th>막는 공격</th></tr></thead>
  <tbody>
    <tr><td><strong>Anycast scrubbing center</strong></td><td>모든 트래픽을 가장 가까운 edge로 흡수 → 정상 트래픽만 origin으로 전달</td><td>L3/L4 volumetric flood (UDP·ICMP·반사 증폭)</td></tr>
    <tr><td><strong>L4 rate limit</strong></td><td>per-IP / per-subnet connection rate cap</td><td>SYN flood, connection exhaustion</td></tr>
    <tr><td><strong>JS challenge · CAPTCHA</strong></td><td>의심 클라이언트에 *자바스크립트 풀이* 또는 CAPTCHA 요구</td><td>봇 HTTP flood, scraping</td></tr>
    <tr><td><strong>WAF rules</strong></td><td>SQLi·XSS·known bad pattern 차단 (OWASP CRS)</td><td>L7 exploit attempt</td></tr>
    <tr><td><strong>L7 rate limit</strong></td><td>per-user / per-API key / per-route cap</td><td>API abuse, credential stuffing</td></tr>
    <tr><td><strong>Bot management</strong></td><td>fingerprint·behavioral analysis로 봇 점수화</td><td>스크래퍼·credential stuffing·인벤토리 hoarding</td></tr>
  </tbody>
</DataTable>

<Callout>
  대표 사례: 2018 GitHub 1.35 Tbps memcached 증폭 공격을 Akamai Prolexic이 Anycast scrubbing으로 흡수. 단일 origin이라면 즉시 다운됐을 트래픽이 전 세계 edge에 분산되어 origin은 *영향 0*. 자세한 BAF(Bandwidth Amplification Factor) 수치는 <a href="/docs/security">공격과 방어</a> 페이지.
</Callout>

<Callout tone="bad">
  **edge challenge의 함정**: JS challenge는 합법 사용자에게도 *FCP 지연 1~3초* 추가. 봇으로 오인된 사용자는 떠나간다. *risk score 임계*를 조심스럽게 튜닝, *crawler allowlist*(Googlebot 등) 유지 필수.
</Callout>

DDoS 방어의 layer 분담 전체 도표와 BGP FlowSpec / RTBH 같은 라우팅 layer 운영은 <a href="/docs/security#ddos-분담">공격과 방어</a> · <a href="/docs/advanced/bgp">BGP</a> 페이지로 cross-link.

## Origin Shielding · Cache Key · Tiered Cache

edge가 콘텐츠를 캐시해도 *origin이 burst를 직격으로 받는 패턴*은 흔합니다. 캐시 설계의 표준 운영 어휘:

<KV>
  <KVItem term="Origin shielding">edge와 origin 사이에 *전용 PoP*를 두어 모든 edge가 그 PoP를 거쳐 origin에 도달. cache miss가 모이는 *중간 캐시 layer* (Cloudflare Argo, Akamai Edge DNS Shielding).</KVItem>
  <KVItem term="Tiered cache">L1(edge PoP) 다음 L2(regional shield) 다음 origin. 글로벌 edge가 cache miss 시 *근처 region*의 L2에 먼저 묻고 그게 미스면 origin. cache hit ratio 1.5~3배 향상.</KVItem>
  <KVItem term="Cache key 설계">URL만이 cache key가 아님. <code>Vary</code> 헤더로 user-agent·언어 분리, Cookie·Authorization 제외, 쿼리 string 정렬: 같은 콘텐츠인데 key가 폭증해 cache 무효화되는 함정의 본진.</KVItem>
  <KVItem term="Stale-while-revalidate (RFC 5861)">만료된 캐시를 *그대로 응답*하면서 비동기로 갱신. user-perceived latency 0 추가, origin 부하 분산.</KVItem>
  <KVItem term="Cache invalidation 전파">tag·path·wildcard purge. Cloudflare·Fastly는 글로벌 전파 약 150ms. CDN별 도구 차이 큼.</KVItem>
</KV>

<Callout tone="bad">
  <strong>Cache key 폭주 함정</strong>: 정렬되지 않은 쿼리 string(<code>?a=1&b=2</code> vs <code>?b=2&a=1</code>)이 각각 다른 key가 되거나, Authorization 헤더가 key에 포함돼 *사용자별 캐시*가 됨: hit ratio 1% 미만으로 추락. <code>Vary: Accept-Encoding</code> 정도만 명시, 나머지는 명시적으로 normalize.
</Callout>

<Eyebrow>실패</Eyebrow>

## 로드밸런서가 안 풀어 주는 문제들

- 썬더링 허드: 캐시 무효화 직후 모든 사용자가 origin으로 몰림 → coalescing, jittered TTL
- retry 폭풍: 5xx에 클라이언트가 동시에 재시도 → exponential backoff + jitter
- 흐름 보존: 서버 추가/제거 시 진행 중 연결을 어떻게 다룰지: connection draining
- 비대칭 라우팅: 트래픽이 다른 경로로 돌아오면 stateful 방화벽이 끊을 수 있음
