## 가상 사설 클라우드: 자기만의 IP 세계

VPC는 RFC 1918 사설 IP 대역을 정의한 격리된 네트워크입니다. 같은 클라우드 안에서도 VPC끼리는 기본적으로 통신이 막혀 있습니다.

<KV>
  <KVItem term="CIDR 블록">VPC 생성 시 정함. 보통 <code>10.0.0.0/16</code> 같은 사설 대역. 나중에 확장은 까다로움.</KVItem>
  <KVItem term="Subnet">VPC를 AZ(Availability Zone) × public/private으로 쪼갬. 보통 <code>10.0.1.0/24</code>식.</KVItem>
  <KVItem term="Route Table">서브넷별 라우팅 결정. 기본은 VPC 내부(<code>10.0.0.0/16 local</code>) + 인터넷 게이트웨이 같은 추가 항목.</KVItem>
  <KVItem term="Internet Gateway (IGW)">VPC에 인터넷 접근을 부여. public 서브넷의 default route 목적지.</KVItem>
  <KVItem term="NAT Gateway">private 서브넷이 outbound 인터넷에 나갈 때. SNAT만: 외부에서 들어오는 건 불가.</KVItem>
  <KVItem term="Egress-only IGW (IPv6)">IPv6용 NAT 대응. IPv6는 NAT가 불필요하지만 outbound만 허용하는 게이트웨이.</KVItem>
</KV>

### Public vs Private 서브넷

갈림길은 라우트 테이블의 default route(0.0.0.0/0)가 어디로 향하느냐입니다.

<PCBox>
  <PC tone="good" title="Public Subnet">
    <ul>
      <li><code>0.0.0.0/0 → IGW</code></li>
      <li>퍼블릭 IP 있는 인스턴스는 인터넷에서 직접 도달 가능</li>
      <li>로드밸런서·bastion이 여기</li>
    </ul>
  </PC>
  <PC tone="good" title="Private Subnet">
    <ul>
      <li><code>0.0.0.0/0 → NAT Gateway</code> (또는 없음)</li>
      <li>인터넷에서 직접 도달 불가</li>
      <li>앱 서버·DB가 여기</li>
    </ul>
  </PC>
</PCBox>

## VPC를 잇는 4가지 방법

<DataTable>
  <thead><tr><th>방식</th><th>특징</th><th>한계</th></tr></thead>
  <tbody>
    <tr><td>VPC Peering</td><td>1:1 직접 연결. 저렴.</td><td>전이 라우팅 불가(A↔B, B↔C라도 A↔C는 별도)</td></tr>
    <tr><td>Transit Gateway</td><td>허브-스포크. 수천 VPC·온프레미스 한 곳에서 라우팅.</td><td>전송량 비용. 리전당 1개.</td></tr>
    <tr><td>PrivateLink (VPC Endpoint Service)</td><td>한 VPC의 NLB를 다른 VPC가 endpoint로 노출. 단방향.</td><td>HTTP/TCP만, UDP 미지원</td></tr>

  </tbody>
</DataTable>

<Callout tone="bad">
  CIDR이 겹치는 두 VPC는 peering이나 TGW로 직접 못 잇습니다.
  해법: PrivateLink로 endpoint만 노출, 또는 한 쪽 NAT.
</Callout>

## 데이터센터와 클라우드 잇기

<KV>
  <KVItem term="VPN (Site-to-Site)">IPsec 터널. 빠르게 구성, 인터넷 경유, 지연·대역폭 가변. ~10 Gbps 한계.</KVItem>
  <KVItem term="Direct Connect / Cloud Interconnect">전용 광회선. 일정한 지연(보통 1~3ms), 대역폭 보장(1·10·100 Gbps). 비싸고 프로비저닝 수 주.</KVItem>
  <KVItem term="SD-WAN">여러 회선(인터넷·MPLS·LTE)을 정책 기반으로 묶음. 비용·복원력 절충.</KVItem>
</KV>

### Direct Connect의 BGP 패턴

전용선이라 해도 결국 BGP 세션을 맺는 일입니다:

<Steps>
  <li>고객 라우터와 클라우드 라우터가 BGP eBGP 세션 수립</li>
  <li>고객이 자기 prefix(예: 10.10.0.0/16)를 광고</li>
  <li>클라우드가 VPC prefix(예: 10.0.0.0/16)를 광고</li>
  <li>VIF(Virtual Interface) 단위로 격리: public VIF(공용 서비스 접근) vs private VIF(VPC 접근)</li>
  <li>장애 대비 두 회선을 다른 location에: BGP가 자동 페일오버</li>
</Steps>

## 인터넷·서비스 접근의 옵션 매트릭스

<DataTable>
  <thead><tr><th>목적</th><th>도구</th><th>특징</th></tr></thead>
  <tbody>
    <tr><td>인스턴스 → 인터넷 (outbound)</td><td>NAT Gateway</td><td>관리형. AZ별로 하나씩 추천(고가용).</td></tr>
    <tr><td>인스턴스 → 인터넷 (저비용)</td><td>NAT instance</td><td>EC2에 직접 구성. 저렴하지만 운영 부담.</td></tr>
    <tr><td>인터넷 → 인스턴스</td><td>Internet Gateway + 퍼블릭 IP/EIP</td><td>방화벽은 SG·ACL로.</td></tr>
    <tr><td>인터넷 → 서비스 (L7)</td><td>ALB</td><td>HTTP/HTTPS·WebSocket·gRPC, 헤더/경로 라우팅.</td></tr>
    <tr><td>인터넷 → 서비스 (L4)</td><td>NLB</td><td>TCP/UDP, 정적 IP, 매우 높은 PPS.</td></tr>
    <tr><td>VPC ↔ AWS 서비스(S3 등)</td><td>VPC Endpoint (Gateway)</td><td>S3·DynamoDB만. 라우트 테이블에 prefix list 자동 추가. 무료.</td></tr>
    <tr><td>VPC ↔ AWS 서비스 (대부분)</td><td>VPC Endpoint (Interface)</td><td>ENI를 VPC에 박음. AWS PrivateLink 기반. 시간당·GB당 과금.</td></tr>
  </tbody>
</DataTable>

<Callout>
  <strong>PrivateLink의 가치</strong>: S3로 가는 트래픽도 인터넷을 안 거치고 AWS 백본 안에서 끝남.
  보안(데이터 외부 노출 X) + 비용(NAT GW data processing 회피) + 지연 모두 이득.
  큰 워크로드라면 NAT 비용이 PrivateLink 비용보다 훨씬 큼: <code>aws s3 cp</code>가 NAT GW 통해 나가면 GB당 4.5¢, S3 endpoint면 0¢.
</Callout>

## Route 53 / Cloud DNS의 라우팅 정책

단순 응답을 넘어선 라우팅:

<KV>
  <KVItem term="Simple">기본. 한 레코드 = 한 응답.</KVItem>
  <KVItem term="Weighted">A에 70%, B에 30%식. canary 배포에 활용.</KVItem>
  <KVItem term="Latency-based">사용자의 DNS 리졸버 위치 기준 가장 빠른 리전 선택.</KVItem>
  <KVItem term="Geolocation">사용자 지역 기준 (legal·compliance에 자주).</KVItem>
  <KVItem term="Failover">primary 헬스체크 실패 시 secondary로.</KVItem>
  <KVItem term="Multi-value">여러 IP 응답 (브라우저 client-side LB).</KVItem>
</KV>

<Callout tone="bad">
  Route 53의 헬스체크 페일오버는 <strong>TTL + 헬스체크 주기</strong> 합만큼 지연. TTL 60s + 체크 30s = 페일오버까지 ~90초.
  진짜 빠른 페일오버는 BGP/Anycast가 필요.
</Callout>

## 클라우드 네트워킹의 함정

- **Data egress**: 인터넷으로 나가는 트래픽: 클라우드의 진짜 수익원. AWS는 GB당 ~9¢, 처음 100GB 무료.
- **AZ 간 트래픽**: 1¢/GB. 사소해 보여도 누적 비용이 커집니다. k8s에서 파드가 다른 AZ의 파드와 통신할 때마다 과금이 붙고, <code>topology.kubernetes.io/zone</code> aware routing으로 줄입니다.
- **NAT Gateway**: 처리되는 모든 GB당 4.5¢. S3·DynamoDB 트래픽을 VPC endpoint로 빼면 비용이 크게 줄어듭니다.
- **PrivateLink**: 시간당 ~1¢ + GB당 1¢. 서비스 노출이 적으면 ALB가 더 쌉니다.
- **Transit Gateway**: VPC attachment 시간당 5¢ + GB당 2¢. 많은 VPC면 가성비 좋지만 적으면 peering이 더 쌉니다.

<Callout>
  비용 최적화는 보통 <strong>"트래픽을 인터넷·NAT GW·TGW로 안 보내기"</strong>의 게임.
  VPC Endpoint·내부 LB·캐싱이 가장 큰 레버.
</Callout>

## 자주 묻는 질문

- "Subnet과 AZ의 관계는?" → AWS는 1 subnet = 1 AZ. GCP는 subnet이 리전 전체.
- "두 VPC가 CIDR이 겹친다. 어떻게 통신?" → peering 불가. PrivateLink로 endpoint만 노출, 또는 NAT/SNAT로 IP 변환.
- "Security Group은 stateful이라는데 정확히 무슨 의미?" → outbound 허용한 흐름의 응답이 자동 허용. conntrack 비유.
- "RDS를 두 VPC에서 쓰고 싶다. 옵션은?" → RDS Proxy + PrivateLink, 또는 peering + private DNS resolution.
- "NAT Gateway가 한 AZ에만 있으면 어떤 위험?" → 그 AZ 장애 시 다른 AZ 파드의 outbound 끊김. 모든 AZ에 NAT GW 두는 게 표준.
- "PrivateLink vs VPC Peering, 언제 어느 걸?" → 단방향 서비스 노출은 PrivateLink. 양방향·여러 서비스는 peering/TGW. 보안 격리가 우선이면 PrivateLink.
- "TGW가 SPOF인가?" → 리전 내 분산 구성이지만 리전 단위 장애는 동일하게 영향. 멀티 리전 운영 시 TGW peering으로 잇거나 Cloud WAN.
