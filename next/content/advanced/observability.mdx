<Eyebrow>패킷</Eyebrow>

## tcpdump로 본 그대로 잡기

커널이 NIC에서 패킷을 직접 캡처합니다. BPF 필터로 관심 트래픽만 추립니다.

```bash
# 80포트로 가는/오는 트래픽, 호스트 1.2.3.4에 한정, 패킷을 파일로
sudo tcpdump -i any -nn -w /tmp/c.pcap 'host 1.2.3.4 and port 80'

# 그 자리에서 헤더 + 페이로드 일부 보기
sudo tcpdump -i any -nnvvXSs0 'tcp port 443 and host api.example.com'

# DNS 질의/응답만
sudo tcpdump -i any -nn 'udp port 53'
```

<Callout>
  프로덕션에서는 패킷이 너무 많아 디스크가 폭증할 수 있습니다.
  <code>-c 1000</code>(N개만), <code>-G 60 -W 10</code>(60초씩 10개 로테이션) 같은 옵션으로 제한하세요.
</Callout>

<Eyebrow>eBPF</Eyebrow>

## 커널 안에서 측정하기

eBPF는 검증된 작은 프로그램을 커널 핫패스에 안전하게 끼워 넣게 해주는 기술입니다. tcpdump보다 훨씬 깊이, 커널이 보는 모든 이벤트를 관측할 수 있습니다.

<Callout tone="bad">
  <strong>"성능 비용 거의 없음"은 과장</strong>. 실측 디테일:
  <ul>
    <li><strong>kprobe 오버헤드</strong>: syscall당 100ns ~ 수 µs. 고빈도 syscall(`io_uring`, `epoll_wait`)에 붙이면 누적 큼.</li>
    <li><strong>ringbuf 드롭</strong>: 이벤트 폭증 시 user space가 못 따라가 드롭: 관측의 *불완전성* 인지 필요.</li>
    <li><strong>verifier 제약</strong>: 루프·복잡도·메모리 접근 제한. 우회하려다 panic 사례.</li>
    <li><strong>CO-RE 호환성</strong>: 커널 버전·BTF 미스매치로 production 노드 panic 가능.</li>
    <li><strong>uprobe</strong>는 kprobe보다 훨씬 비쌈 (~µs ~ ms).</li>
  </ul>
  <strong>저비용이지 무비용 아님</strong>. 고빈도 hot path에 무차별 부착 금지.
</Callout>

```bash
# BCC/bpftrace 유틸들
sudo tcpconnect-bpfcc          # 신규 TCP 연결 실시간 로그
sudo tcpretrans-bpfcc          # 재전송 발생 위치·시간
sudo tcpaccept-bpfcc           # listen 측 수락 이벤트
sudo tcplife-bpfcc             # 연결 수명·바이트 수
sudo gethostlatency-bpfcc      # DNS 조회 지연 분포
sudo execsnoop-bpfcc           # 어떤 프로세스가 어떤 명령을 실행했나
```

<Callout tone="good">
  eBPF는 "왜 갑자기 99분위 응답이 늘었지?", "어떤 클라이언트가 재전송을 유발하지?" 같은 질문에 강력합니다.
  Cilium·Pixie·Coroot 같은 도구들이 이를 활용해 관측·네트워킹·보안을 한 번에 다룹니다.
</Callout>

<Eyebrow>Service Mesh</Eyebrow>

## 모든 마이크로서비스 호출을 가로채는 사이드카

Istio·Linkerd·Consul은 각 파드 옆에 사이드카 프록시(주로 Envoy)를 배치합니다.
앱의 모든 인바운드/아웃바운드 트래픽이 이 프록시를 거치기 때문에, 앱 코드를 바꾸지 않고도 다음을 얻습니다:

- 자동 mTLS: 서비스 간 통신이 자동 암호화되고 신원이 검증됩니다
- 관측: 모든 호출의 RPS·p99·에러율이 자동 메트릭으로 노출됩니다
- 분산 트레이싱: trace ID 자동 전파, span 자동 생성
- 트래픽 정책: 카나리 배포 5%·재시도·서킷브레이커·rate limit를 YAML 한 줄로
- 인가: "service-a만 service-b를 호출 가능" 같은 ACL

<Callout tone="bad">
  <strong>사이드카 비용을 숫자로</strong>:
  <ul>
    <li>파드당 메모리 <strong>50~150MB</strong> (Envoy 기본 100MB+, Linkerd-proxy ~50MB)</li>
    <li>p99 지연 <strong>+1~3ms</strong> 추가 hop (TLS handshake + 정책 평가)</li>
    <li>iptables redirection 디버깅 지옥 (istio-init의 `15001`/`15006` 리다이렉트)</li>
    <li>Envoy <strong>config push storm</strong> (서비스 수천 개에서 endpoint 변경 시 모든 사이드카 재컨피그)</li>
    <li>mTLS 인증서 자동 회전 실패 시 전체 통신 마비</li>
  </ul>
  대안: <strong>ambient mesh</strong> (Istio Ambient, 2023~) 또는 <strong>eBPF 기반</strong> (Cilium Service Mesh): 사이드카 없이 노드 단 프록시. 트레이드오프: L7 정책 일부 제한.
</Callout>

<Callout tone="bad">
  <strong>mTLS와 패킷 캡처의 충돌</strong>. 사이드카가 mTLS를 자동 적용하면 *노드 외부 캡처*는 모두 암호문. 디버깅 우회:
  <ul>
    <li>**파드 안 localhost 캡처** (사이드카 → 앱 컨테이너 사이는 평문)</li>
    <li>**SSLKEYLOGFILE** + Wireshark의 TLS 복호화</li>
    <li>**eBPF의 `ssl_read`/`ssl_write` uprobe** (OpenSSL 함수 직접 후킹, BoringSSL은 별도)</li>
    <li>Cilium **Hubble**: eBPF 단에서 평문 가시화</li>
  </ul>
</Callout>

## 요청 하나의 인생: 분산 트레이싱

모놀리식이 마이크로서비스로 쪼개진 뒤 "이 요청이 왜 느리지?"는 경계를 가로지르는 질문이 됐습니다.
해법은 trace ID 전파: 각 호출이 같은 trace ID + 부모 span ID를 가지면, 끝에서 한 그래프로 재조립할 수 있습니다.

```
# W3C Trace Context (HTTP 헤더로 자동 전파)
traceparent: 00-{trace-id 32hex}-{span-id 16hex}-01
tracestate: rojo=00f067aa0ba902b7,congo=t61rcWkgMzE
```

OpenTelemetry가 사실상 표준입니다. SDK가 자동으로 헤더 전파·span 생성을 처리하고,
백엔드는 Jaeger·Tempo·Honeycomb 등 선택. 중요 지표 4가지(USE·RED·Golden Signals)를 SLO와 연결해 알람으로 잇습니다.

### 샘플링 전략: 핵심 운영 결정

분산 트레이싱은 *모든 요청 저장*이면 storage가 폭주합니다 (Jaeger·Tempo가 가장 빨리 죽는 이유). 샘플링이 필수:

<KV>
  <KVItem term="Head-based sampling">트레이스 시작 시점에 1%/10%/100% 같은 비율로 결정. <strong>장점</strong>: 일관성·구현 단순. <strong>단점</strong>: 희귀 에러를 놓침 (1% 샘플링이면 1000개 중 1개 에러도 1/100 확률로만 잡힘).</KVItem>
  <KVItem term="Tail-based sampling">트레이스 *완성 후* 의사결정 (에러 있으면 100%, 느린 요청만, 또는 RPC 패턴별). <strong>장점</strong>: 가치 있는 트레이스만 저장. <strong>단점</strong>: 트레이스 전체를 메모리에 버퍼링 필요 → OpenTelemetry Collector tail sampler 같은 전용 컴포넌트.</KVItem>
  <KVItem term="카디널리티 폭발">메트릭 라벨에 `user_id`·`request_id` 박으면 시계열 폭증 → Prometheus 다운. <strong>고-카디널리티는 trace로</strong>, low-카디널리티는 metric으로 분리가 원칙.</KVItem>
  <KVItem term="Exemplars">메트릭 점에서 trace로 점프하는 OpenTelemetry 기능. p99 spike 점에서 그 시각의 실제 trace ID로 한 클릭.</KVItem>
</KV>

<KV>
  <KVItem term="RED">Rate · Errors · Duration: 서비스 단위 외부 관점</KVItem>
  <KVItem term="USE">Utilization · Saturation · Errors: 리소스(CPU·메모리·디스크) 관점</KVItem>
  <KVItem term="Golden Signals">Latency · Traffic · Errors · Saturation: Google SRE</KVItem>
</KV>

<Eyebrow>Golden Signals · MTTR</Eyebrow>

## 무엇을 측정하나: 얼마나 빨리

관측의 *목적*은 <strong>SLO</strong>(Service Level Objective = 신뢰성 목표, 예: p99 ≤ 200ms를 30일 평균 99.9%) 추출·<strong>MTTR</strong>(Mean Time To Recovery = 평균 복구 시간) 단축. SLO·SLI의 정의·운영 결정은 <a href="/docs/advanced/reliability">신뢰성 패턴</a> 페이지가 본진이고, 본 페이지는 *측정 구현* 분담.

표준 신호 모델 3종:

<DataTable>
  <thead><tr><th>모델</th><th>출처</th><th>지표</th><th>적합</th></tr></thead>
  <tbody>
    <tr><td><strong>Four Golden Signals</strong></td><td>Google SRE</td><td>Latency · Traffic · Errors · Saturation</td><td>request·response 서비스 (HTTP·gRPC)</td></tr>
    <tr><td><strong>RED method</strong></td><td>Tom Wilkie (Weaveworks)</td><td>Rate · Errors · Duration</td><td>마이크로서비스 (golden signals 단순화)</td></tr>
    <tr><td><strong>USE method</strong></td><td>Brendan Gregg</td><td>Utilization · Saturation · Errors</td><td>하드웨어·OS 자원 (CPU·디스크·NIC)</td></tr>
  </tbody>
</DataTable>

<KV>
  <KVItem term="MTTD (Detect)">사고가 *발생*하고 *알람이 울리기까지*. 관측이 이걸 직접 결정. *유저 영향 시각 - 알람 시각*. 분~시간 단위면 관측 부족 신호.</KVItem>
  <KVItem term="MTTR (Recovery)">알람 후 *복구까지*. runbook·자동화·rollback 능력이 결정.</KVItem>
  <KVItem term="MTBF (Between Failures)">두 사고 사이 *정상 시간*. 안정성 trend KPI.</KVItem>
  <KVItem term="Time to Investigate">알람 후 *원인 파악*까지. trace·log·exemplars의 가치가 이 지표에 직접 반영.</KVItem>
</KV>

<Callout>
  <strong>분담 명시</strong>: *이 페이지*가 *측정 구현* (SLI 추출·exemplars·trace·MTTR 측정 도구). <a href="/docs/advanced/reliability">신뢰성 패턴</a>이 *정의·운영 결정* (SLO·error budget·burn rate·DORA).
</Callout>

## 장애가 났을 때: 추적 순서

<Steps>
  <li><strong>지표·대시보드</strong>로 어떤 서비스가 비정상인지 추정 (RED, Golden Signals)</li>
  <li><strong>트레이스</strong>로 그 서비스 안 어느 호출이 느린지/실패하는지 추적</li>
  <li><strong>로그</strong>로 해당 트레이스 ID의 컨텍스트를 확인: 에러 메시지·스택</li>
  <li>네트워크 의심 시 <strong>eBPF</strong>로 커널 관점 검증: 재전송·연결 거절·DNS 지연</li>
  <li>마지막 결정타로 <strong>tcpdump·Wireshark</strong>: 패킷 그 자체</li>
</Steps>

<Callout tone="good">
  "지표 → 트레이스 → 로그 → 커널 이벤트 → 패킷"의 순서는 <strong>가설을 좁히고, 비싼 도구는 늦게</strong>라는 원칙의 응용입니다.
  평소 관측 인프라를 갖춰 두면 새벽 3시에 길을 잃지 않습니다.
</Callout>
