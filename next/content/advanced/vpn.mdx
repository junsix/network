## 패킷을 다른 패킷 안에 넣기

## 전통의 표준

IPsec은 단일 프로토콜이 아니라 모음입니다.

<KV>
  <KVItem term="IKE (v2)">키 교환·인증·정책 협상. UDP/500, NAT 통과 시 UDP/4500.</KVItem>
  <KVItem term="ESP">실제 데이터 보호. 암호화 + 무결성 (요즘은 AES-GCM 같은 AEAD).</KVItem>
  <KVItem term="AH">무결성만 (암호화 없음). 거의 안 씀.</KVItem>
  <KVItem term="Transport mode">원본 IP 헤더 유지, payload만 보호. 호스트-호스트.</KVItem>
  <KVItem term="Tunnel mode">원본 IP까지 통째로 캡슐화. 게이트웨이-게이트웨이.</KVItem>
</KV>

<Callout>
  IPsec은 강력하지만 구성이 복잡하고 옵션이 매우 많아 오설정이 잦습니다.
  NAT·방화벽과의 상호작용도 까다롭습니다 (그래서 NAT-T가 별도 표준으로 등장).
</Callout>

## 현대적 단순함

WireGuard는 "필요한 것만 남기자"는 철학으로 다시 짠 VPN입니다.
암호 스위트는 단 하나로, Curve25519(키 교환), ChaCha20-Poly1305(AEAD), BLAKE2(해시), SipHash(테이블)를 씁니다.
커널 코드 약 4,000줄로 IPsec의 수십만 줄 대비 감사할 수 있는 규모입니다.

<MetricRow>
  <Metric value="~4K" label="커널 코드 라인" />
  <Metric value="1" label="암호 스위트" />
  <Metric value="UDP only" label="전송" accent />
  <Metric value="stateless" label="서버 메모리 부담" />
</MetricRow>

```ini
[Interface]
PrivateKey = aBcDeF...
Address    = 10.0.0.2/32

[Peer]
PublicKey  = 1234ABCD...
AllowedIPs = 10.0.0.0/24
Endpoint   = vpn.example.com:51820
PersistentKeepalive = 25
```

설정이 이 정도로 끝납니다. 키 교환은 노이즈(Noise) 프로토콜 프레임워크를 따릅니다.

## onion routing: 신원 자체를 가리기

VPN은 "내가 어디로 가는지를 ISP에게 가린다"이지만,
Tor는 한 단계 더 나아가 "도착지도 출발지를 모르게" 합니다.
패킷을 양파 껍질처럼 세 겹으로 감싸, 각 노드가 자기 다음 노드만 알 수 있도록 전달합니다.

<Steps>
  <li><strong>Entry (Guard)</strong>: 사용자 IP를 안다. 다음이 어디인지는 모름.</li>
  <li><strong>Middle</strong>: 사용자도, 도착지도 모름. Entry와 Exit만 안다.</li>
  <li><strong>Exit</strong>: 도착지를 안다. 사용자 IP는 모름. (이 노드가 평문을 본다: HTTPS 필수)</li>
</Steps>

<Callout tone="bad">
  Exit 노드는 트래픽을 그대로 인터넷에 내보냅니다. HTTPS가 아니면 도청 가능합니다.
  또한 글로벌 관찰자가 Entry·Exit 양쪽 트래픽 패턴을 비교하면 (correlation) 익명성이 깨질 수 있습니다.
</Callout>

## VPN이 풀지 못하는 것

- **엔드포인트 보안**: 단말이 감염되면 VPN으로 사내망에 침투 경로 제공
- **지나친 신뢰**: "VPN으로 접속했으니 사내인 척" 가정 → 측면 이동 쉬움 (Zero Trust가 등장한 이유)
- **DNS 누수**: VPN 활성 중에도 OS가 평소 리졸버에 묻는 경우
- **운영자 신뢰**: 상용 VPN 사업자는 ISP를 또 다른 ISP로 대체하는 효과: 로그 정책 검증 어려움
