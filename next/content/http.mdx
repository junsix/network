## 요청과 응답의 모양

```
GET /index.html HTTP/1.1
Host: example.com
User-Agent: curl/8.4.0
Accept: */*
```

```
HTTP/1.1 200 OK
Content-Type: text/html; charset=utf-8
Content-Length: 1256
Cache-Control: max-age=3600

<!doctype html> ...
```

첫 줄(요청 라인 또는 상태 라인), 헤더, 빈 줄, 본문. 텍스트라 어떤 도구로도 들여다보기 쉽습니다.
이 단순함이 HTTP의 성공 비결이지만, 이후 성능 문제의 근원이기도 합니다.

## HTTP 위에 TLS

HTTPS는 별도의 프로토콜이 아닙니다. HTTP를 **TLS** 위에서 전송하는 것뿐입니다.
그래서 도청·위변조·신원 사칭에서 자유로워지지만, 핸드셰이크 비용과 인증서 운영 부담이 생깁니다.

자세한 TLS 핸드셰이크는 [TLS 페이지](/docs/tls)에서 다룹니다.

## 캐싱: ETag, Vary, stale-while-revalidate

HTTP 캐싱은 단순히 "TTL"이 아닙니다. **freshness** 와 **validation** 두 개의 메커니즘이 있고, 둘이 같이 동작합니다.

### Cache-Control 디렉티브 풀세트

<KV>
  <KVItem term="max-age=N"><strong>private 캐시</strong>(브라우저)의 freshness. N초간 origin에 안 묻고 그대로 사용.</KVItem>
  <KVItem term="s-maxage=N"><strong>shared 캐시</strong>(CDN/proxy)에만 적용. 보통 max-age보다 크게: 브라우저는 짧게, 엣지는 길게.</KVItem>
  <KVItem term="no-cache">매번 검증 필수 (그러나 캐시는 보관). 검증 후 304 받으면 캐시 본문 재사용.</KVItem>
  <KVItem term="no-store">아예 저장 금지. 민감 데이터(인증·결제) 응답에.</KVItem>
  <KVItem term="must-revalidate">stale 상태에서는 반드시 origin 검증. 네트워크 끊기면 504 반환.</KVItem>
  <KVItem term="stale-while-revalidate=N">만료된 응답을 N초간 그대로 주면서 백그라운드에서 갱신. <strong>체감 속도의 핵심.</strong></KVItem>
  <KVItem term="stale-if-error=N">origin이 5xx 줄 때 N초까지는 만료된 캐시로 응답. 안정성 패턴.</KVItem>
  <KVItem term="immutable">URL 자체가 버전을 포함(예: <code>app.abc123.js</code>) → 영원히 같은 콘텐츠. 검증조차 안 함.</KVItem>
</KV>

### Validation: 조건부 요청

만료된 응답을 다시 쓸 수 있는지 묻는 두 방식:

- **ETag**: 콘텐츠의 해시·버전. 요청 시 <code>If-None-Match: "abc123"</code>. 같으면 **304 Not Modified** + 본문 0바이트.
- **Last-Modified**: 마지막 수정 시각. <code>If-Modified-Since: ...</code>. ETag보다 정밀도 낮음.

### Vary: 같은 URL의 여러 표현

<code>Vary: Accept-Encoding</code>이면 gzip 사용자와 brotli 사용자에게 다른 응답: 캐시 키에 헤더가 포함됨.
<code>Vary: Cookie</code>는 캐시 *키*가 쿠키 값마다 분리됨 (저장은 됨, 다만 사용자마다 키가 달라 사실상 사용자별 캐시 = hit ratio 폭락). 무효화(no-store)와는 다름.

<Callout tone="bad">
  <strong>CDN 운영 함정</strong>: <code>Vary: User-Agent</code>는 거의 모든 UA마다 다른 캐시 항목 →
  히트율 0%에 가까워짐. 클라이언트 힌트(<code>Sec-CH-UA</code>)나 명시적 변형 사용.
</Callout>

## CORS: 브라우저의 동일출처 정책 우회

다른 origin(<code>https://a.com</code> → <code>https://api.b.com</code>)으로의 fetch는 기본 차단. **CORS**가 origin 간 명시적 허용 메커니즘:

### Simple request vs Preflight

- **Simple**: GET/POST/HEAD + CORS-safelisted 헤더만(Accept/Accept-Language/Content-Language + Content-Type은 form 계열만) → 그냥 요청 보내고 응답의 <code>Access-Control-Allow-Origin</code> 검사.
- **Preflight**: PUT/DELETE/PATCH, 또는 커스텀/제한 헤더 (`Authorization` 포함, `X-*` 커스텀, `application/json` Content-Type 등) → 실 요청 전 **OPTIONS 요청** 으로 허용 여부 먼저 확인. 비용 1 RTT 추가. 운영에서 `Authorization` 헤더 때문에 preflight 발동하는 게 가장 흔함.

```http
OPTIONS /api/items HTTP/1.1
Origin: https://a.com
Access-Control-Request-Method: DELETE
Access-Control-Request-Headers: x-csrf-token

# 응답
HTTP/1.1 204 No Content
Access-Control-Allow-Origin: https://a.com
Access-Control-Allow-Methods: GET, POST, DELETE
Access-Control-Allow-Headers: x-csrf-token
Access-Control-Max-Age: 86400   ← preflight 캐시 24시간
```

<Callout tone="bad">
  <code>Access-Control-Allow-Origin: *</code>와 <code>Access-Control-Allow-Credentials: true</code>를 같이 쓰면 브라우저가 거부.
  쿠키를 동반하려면 origin을 명시해야 함.
</Callout>

### CORS vs CSRF

자주 혼동하는 두 개념:
- **CORS**: 다른 origin의 응답을 *읽을 수 있는가*. 브라우저가 강제.
- **CSRF**: 다른 origin이 *내 쿠키로 요청을 보낼 수 있는가*. 서버가 토큰·SameSite로 막아야 함.
CORS는 CSRF를 자동으로 막아주지 *않습니다*.

## 쿠키 보안: 2024+ 모범

<KV>
  <KVItem term="HttpOnly">JavaScript가 못 읽음. XSS로 토큰 탈취 차단.</KVItem>
  <KVItem term="Secure">HTTPS에서만 전송. HTTP 다운그레이드 노출 방지.</KVItem>
  <KVItem term="SameSite=Strict">크로스 사이트 요청에 쿠키 미전송. CSRF 차단 + 외부 링크로 들어올 때도 로그아웃 상태로 보임.</KVItem>
  <KVItem term="SameSite=Lax">기본값. 톱-레벨 navigation(링크 클릭)엔 전송, iframe·서드파티 fetch엔 미전송.</KVItem>
  <KVItem term="SameSite=None">크로스 사이트 허용. Secure 필수.</KVItem>
  <KVItem term="__Host- prefix">쿠키 이름이 <code>__Host-</code>로 시작하면 브라우저가 강제: Secure + Path=/ + Domain 없음. <strong>가장 안전한 쿠키 패턴.</strong></KVItem>
  <KVItem term="Partitioned (CHIPS)">서드파티 쿠키 차단 시대의 절충안. 임베드된 사이트가 자기 파티션의 쿠키만 사용.</KVItem>
</KV>

## 보안 헤더: 최소 셋업

```http
Strict-Transport-Security: max-age=31536000; includeSubDomains; preload
Content-Security-Policy: default-src 'self'; script-src 'self' 'nonce-abc123'
X-Content-Type-Options: nosniff
Referrer-Policy: strict-origin-when-cross-origin
Permissions-Policy: geolocation=(), microphone=()
```

- **HSTS**: HTTP → HTTPS 강제. <code>preload</code>를 붙이고 hstspreload.org에 등록하면 브라우저 내장 목록에 들어가 첫 방문부터 HTTPS 강제.
- **CSP nonce**: 인라인 스크립트를 무작위 nonce와 묶어 XSS 방어. <code>'unsafe-inline'</code>은 마지막 선택지.

## HTTP/2·3에서 변경되는 것들

- **헤더 압축**: 같은 헤더가 반복되면 HPACK/QPACK이 거의 0바이트. → "더 많은 헤더 = 느림"이 더 이상 안 맞음.
- **Server Push**: HTTP/2에서 도입됐다 사실상 폐기 (Chrome이 제거). 이제는 **103 Early Hints**가 그 자리를 차지.
- **Prioritization**: HTTP/2의 dependency tree는 복잡해서 잘 안 됐고, HTTP/3는 **단순한 우선순위 헤더**로 회귀.
- **CONNECT-UDP / MASQUEUE**: HTTP/3 위에서 UDP/IP 터널링. VPN 차단 우회 시도에 활용.
