## 브라우저에서 외부 리졸버까지: OS 단계

"DNS 조회"는 한 번에 일어나지 않습니다. 브라우저가 외부 리졸버에 닿기 전까지 OS는 다음 순서를 거칩니다.

<Steps>
  <li><strong>브라우저 in-memory 캐시</strong>: Chrome은 약 1분간 자체 캐시. <code>chrome://net-internals/#dns</code></li>
  <li><strong>OS 스텁 리졸버 캐시</strong>: Linux는 systemd-resolved/nscd, macOS는 mDNSResponder, Windows는 DNS Client 서비스</li>
  <li><strong>/etc/hosts</strong>: 정적 매핑. Kubernetes 같은 곳에서 <code>HostAliases</code>로 주입</li>
  <li><strong>NSS (Name Service Switch)</strong>: <code>/etc/nsswitch.conf</code>의 <code>hosts:</code> 라인이 결정. 보통 <code>files dns</code> 순서. mDNS, LDAP도 가능</li>
  <li><strong>mDNS</strong>: <code>.local</code> 도메인은 멀티캐스트로 LAN에 질의 (DNS 안 거침)</li>
  <li><strong>외부 리졸버</strong>: <code>/etc/resolv.conf</code>의 <code>nameserver</code> (대개 ISP/DHCP가 주입한 주소)</li>
</Steps>

<Callout tone="bad">
  <strong>운영 함정</strong>: 컨테이너에서 <code>/etc/resolv.conf</code>의 <code>search</code> 도메인이 길면 한 번의 외부 조회가 N+1번이 됨 (Kubernetes의 유명한 "5초 DNS 지연" 버그).
  K8s에서는 <code>ndots:5</code>가 기본인데, FQDN(끝에 점 붙임)을 쓰면 search 확장을 건너뜀.
</Callout>

## 512바이트와 EDNS

원래 DNS over UDP는 **응답 크기 512바이트** 제한이 있었습니다. 넘으면 응답에 TC(truncated) 비트가 켜져서 클라이언트가 TCP로 재시도. 그러나 IPv6·DNSSEC·SRV 등으로 응답이 자주 512를 넘게 되면서 **EDNS(0)** 가 등장:

- 클라이언트가 OPT 레코드로 "나는 4096바이트까지 받을 수 있어"를 광고
- 서버가 그만큼 보내면 UDP fragmentation 없이 큰 응답 전달
- 다만 4096은 너무 커서 **IP fragmentation 공격 표면**이 되어 요즘은 **1232바이트**(IPv6 최소 MTU - 헤더)를 권장 ("DNS Flag Day 2020")

<Callout tone="bad">
  <strong>왜 4096이 위험한가: off-path fragmentation 공격</strong>. 큰 DNS 응답이 IP 단편화되면 첫 조각만 보고 후속 조각을 위조하는 *off-path injection*이 가능(Shulman, DNSpooq 2021). 따라서 EDNS bufsize를 1232로 내리고, 1232 초과 응답은 **TC bit로 TCP fallback** 유도가 표준. 사내 리졸버에 1232 설정이 되어 있는지 점검은 시니어 운영의 기본.
</Callout>

## Glue record: 닭과 달걀

<code>example.com</code>의 권한 서버가 <code>ns1.example.com</code>이라면, 그 이름을 풀려고 또 <code>example.com</code>을 물어야 함 → 무한 루프.

해결: 부모 존(<code>.com</code> TLD)이 권한 위임을 응답할 때 NS 이름과 **그 IP까지 함께** 추가 섹션에 끼워 넣음. 이 IP가 **glue record**. 권한 서버 이름이 자기 도메인 안에 있을 때만 필요(in-bailiwick).

<Callout tone="bad">
  <strong>glue stale 위험</strong>. 부모 존의 glue와 자식 존의 실제 A 레코드가 어긋나면 리졸버마다 다른 IP를 봄. 더 골치 아픈 점: <strong>BIND vs Unbound 동작이 다름</strong> (BIND는 glue 우선 신뢰, Unbound는 자식 존 권한 응답으로 갱신 시도). 2021 Slack 부분 장애, 2016 Dyn 사건의 핵심 토픽. 운영 점검: <code>dig +trace</code>로 부모 ↔ 자식 NS·glue 일치성 정기 확인.
</Callout>

## Happy Eyeballs: IPv4 vs IPv6 우선순위

호스트가 A와 AAAA 둘 다 받았을 때 어느 쪽으로 먼저 연결할지가 문제입니다. IPv6를 우선했더니 망가진 v6 경로 때문에 사용자 화면이 멈춰버린 사건이 많았죠.

**Happy Eyeballs (RFC 8305)** 의 답: 두 주소에 *거의 동시에* TCP를 시작해서 먼저 핸드셰이크 끝나는 쪽 채택. v4를 v6보다 50ms 늦게 시작해 살짝 v6 우선: 망가진 v6면 빠르게 v4로 폴백.

## 없는 이름도 캐시한다

<code>NXDOMAIN</code> 응답도 캐시됩니다. SOA 레코드의 마지막 필드 <code>minimum</code>(또는 응답 자체 TTL 중 작은 값)이 그 기간. **존재하지 않는 이름을 자꾸 묻는 봇 트래픽**을 권한 서버까지 흘러가지 않게 막아주는 장치.

## TTL 설계의 트레이드오프

<DataTable>
  <thead><tr><th>TTL</th><th>장점</th><th>단점</th></tr></thead>
  <tbody>
    <tr><td>짧음 (30s~5m)</td><td>장애 시 빠른 페일오버, 트래픽 이동 즉시 반영</td><td>리졸버 부하 ↑, 권한 서버 비용 ↑, BGP·CDN 안정성 의존 ↑</td></tr>
    <tr><td>긴 (1d~7d)</td><td>저비용, 안정</td><td>장애 시 회복까지 TTL만큼 대기, 마이그레이션 어려움</td></tr>
  </tbody>
</DataTable>

<Callout>
  실무 절충: 평소엔 길게(예: 1시간), 마이그레이션 *직전*에 짧게(60초) 내려두고 작업 후 다시 올림.
  공중 리졸버는 종종 권한 TTL을 무시하고 자기 정책으로 캐시함을 기억할 것 (Google, Cloudflare 등).
</Callout>

<Callout tone="bad">
  <strong>TTL이 짧다고 페일오버가 빠른 건 아니다</strong>. 흔히 빠지는 함정:
  <ul>
    <li><strong>리졸버 TTL floor 강제</strong>: 일부 ISP/공중 리졸버는 *너무 짧은 TTL*에 하한(예: 60s)을 강제. 권한 서버에 30s 박아도 60s로 캐시됨.</li>
    <li><strong>브라우저·OS·앱 자체 캐시</strong>: Chrome ~60s, Java InetAddress는 기본 영구 캐시 (<code>networkaddress.cache.ttl</code> 명시 안 하면), Go/Python도 라이브러리별 다름.</li>
    <li><strong>negative cache</strong>: NXDOMAIN도 SOA <code>minimum</code> 동안 캐시 → 새 레코드 추가 직전에 NXDOMAIN 캐시되면 그 시간만큼 invisible.</li>
    <li><strong>커넥션 풀 재사용</strong>: DNS는 갱신돼도 살아있는 HTTP/2 연결은 옛 IP 그대로 사용.</li>
  </ul>
  진짜 빠른 페일오버는 BGP Anycast 또는 LB의 health check + connection draining 조합.
</Callout>
