## 핵심 한 줄

<Callout tone="good">
  공개키로 키 교환을 안전하게 협상한 뒤, 실제 데이터는 빠른 대칭키로 암호화한다.

공개키 암호는 인증·키 교환에는 강하지만 느립니다. 대칭키는 빠르지만 둘이 같은 키를 가지고 있어야 합니다.
TLS 핸드셰이크는 "느린 공개키로 빠른 대칭키를 안전하게 합의"하는 절차입니다.

## 왜 디피-헬만(DH)을 쓰나: 순방향 비밀성

과거 RSA 키 교환에서는 서버 개인키만 알면 과거 트래픽까지 복호화할 수 있었습니다.
오늘날에는 매 세션마다 임시 ECDHE 키를 만들어, 세션 종료 후엔 그 키가 사라집니다.
**나중에 서버 개인키가 유출돼도 과거 트래픽은 복호화 불가**: 이것이 PFS입니다.

## 면접관이 파고드는 곳

### SNI: 같은 IP, 여러 사이트

ClientHello에는 **Server Name Indication(SNI)** 확장이 들어갑니다: "내가 접속하려는 도메인은 `api.example.com`이다". 서버는 이걸 보고 어떤 인증서를 제시할지 결정.

<Callout tone="bad">
  SNI는 <strong>평문</strong>입니다. 도청자는 "어디로 가는지"를 정확히 알 수 있음.
  이걸 푸는 게 <strong>ECH (Encrypted Client Hello)</strong>: Cloudflare·Firefox·Chrome이 점진 배포 중.
  ECH 키는 DNS의 HTTPS 레코드로 전달.
</Callout>

### Session resumption: 1-RTT를 0-RTT 가까이

같은 서버에 재방문할 때 매번 전체 핸드셰이크를 다시 하는 건 낭비입니다. TLS 1.3은 두 가지 재개 방식을 제공:

<DataTable>
  <thead><tr><th>방식</th><th>저장 위치</th><th>특징</th></tr></thead>
  <tbody>
    <tr><td>Session ID</td><td>서버 (메모리/캐시)</td><td>TLS 1.2 시대. 서버 스케일아웃 시 sticky 필요</td></tr>
    <tr><td>Session Ticket</td><td>클라이언트가 보관 (서버가 발급한 암호화 블롭)</td><td>서버 stateless 가능. 키 회전 중요</td></tr>
    <tr><td>PSK (TLS 1.3)</td><td>위 둘을 통합한 모델</td><td>0-RTT 데이터 동봉 가능</td></tr>
  </tbody>
</DataTable>

### 인증서 폐기: 어렵게 푸는 문제

만료 전에 인증서를 무효화하려면? 발급된 인증서는 이미 클라이언트가 신뢰함. 폐기는 **신뢰 깨기** 작업이라 본질적으로 어렵습니다.

<KV>
  <KVItem term="CRL">발급 CA가 폐기 목록 게시. 크고 느림: 사실상 사장.</KVItem>
  <KVItem term="OCSP">매 검증마다 CA의 OCSP 서버에 묻기. 사용자 추적 우려 + CA 장애 시 fail-open 관행.</KVItem>
  <KVItem term="OCSP Stapling">서버가 자기 인증서의 OCSP 응답을 미리 받아 핸드셰이크에 끼워 넣음. 가장 실용적.</KVItem>
  <KVItem term="Must-Staple">인증서에 "OCSP staple 필수" 플래그. fail-open 막음.</KVItem>
  <KVItem term="CRLite (Firefox)">CRL 전체를 Bloom filter로 압축해 브라우저가 내장. 오프라인 검증 가능.</KVItem>
  <KVItem term="짧은 만료">2024년부터 90일 → 47일 → 향후 단축 추세. "폐기보다 갱신으로 풀자".</KVItem>
</KV>

### ALPN: 어떤 상위 프로토콜?

같은 443 포트로 HTTP/1.1·HTTP/2·HTTP/3 후보가 다 있을 때, 핸드셰이크 중 **ALPN(Application-Layer Protocol Negotiation)** 으로 협상합니다:

```
ClientHello: ALPN = ["h2", "http/1.1"]
ServerHello: ALPN = "h2"   ← 서버가 선택
```

HTTP/3는 TLS가 QUIC 안에 통합돼 있어 ALPN으로 `h3` 협상. 그래서 같은 도메인이 1.1/2/3을 다 지원할 수 있음.

### 인증서 투명성 (CT)

CA가 잘못 발급해도 클라이언트는 모르는 문제. **CT 로그**는 모든 발급 인증서를 공개 append-only 로그에 박아 누구나 감시 가능. 브라우저는 인증서에 **SCT(Signed Certificate Timestamp)** 가 없으면 거부.
당사자(예: 우리 회사)는 <code>crt.sh</code> 같은 서비스로 "내 도메인에 누가 인증서 발급했나"를 감시.

### TLS termination vs Passthrough

로드밸런서에서 TLS를 어떻게 다룰지:

<PCBox>
  <PC tone="good" title="Termination (LB에서 복호화)">
    <ul>
      <li>L7 라우팅 가능: URL/헤더 기반 분기</li>
      <li>WAF·캐시 적용 가능</li>
      <li>인증서 운영이 한 곳으로 집중</li>
      <li>LB ↔ 백엔드는 평문 또는 내부 TLS 재암호화</li>
    </ul>
  </PC>
  <PC tone="good" title="Passthrough (TCP 그대로 전달)">
    <ul>
      <li>LB는 SNI만 보고 백엔드 선택 (SNI-based routing)</li>
      <li>End-to-end 암호화 보장 (LB도 못 봄)</li>
      <li><strong>mTLS</strong>(mutual TLS, 서버·클라이언트 양쪽이 인증서로 신원 증명) 필요한 워크로드 (금융·헬스)에 적합. 자세히는 <a href="/docs/advanced/auth">OAuth · OIDC</a> 페이지.</li>
      <li>L7 기능 포기</li>
    </ul>
  </PC>
</PCBox>

### Post-Quantum: 이미 시작됨

양자 컴퓨터가 등장하면 ECDHE/RSA 키 교환은 깨집니다. NIST가 **Kyber(ML-KEM)** 를 표준화했고, Chrome·Cloudflare는 2024년부터 **X25519MLKEM768 하이브리드**(고전 + PQ를 같이 사용)를 점진 배포 중. 핸드셰이크 메시지가 1KB 정도 커져서 미들박스 호환성이 화제.

<Callout tone="bad">
  <strong>"지금 캡처해서 미래에 복호화" (Harvest Now, Decrypt Later)</strong> 공격이 PQ 마이그레이션의 진짜 동인.
  민감 데이터는 이미 오늘 PQ로 보호해야 10~20년 뒤 안전.
</Callout>
