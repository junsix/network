<Eyebrow>왜 DHCP가 먼저인가</Eyebrow>

## ARP·DNS·HTTP는 IP가 있어야 동작한다

PC가 처음 LAN에 붙으면 자기 IP가 없습니다. ARP로 다른 호스트 MAC을 찾으려 해도 *자기 IP*가 비어 있고, DNS 서버에 질의를 보내려 해도 *DNS 서버 IP*를 모릅니다. DHCP가 이 둘을 동시에 해결합니다: **자기 IP + 게이트웨이 + DNS 서버 + 도메인 + (옵션) NTP·MTU·검색 목록**.

<Steps>
  <li><strong>Discover</strong>: src 0.0.0.0, dst 255.255.255.255. PC는 아직 IP가 없어 source가 0이고 broadcast로 보냅니다.</li>
  <li><strong>Offer</strong>: 서버가 풀에서 후보 IP를 잡아 응답. 여러 서버가 있으면 여러 Offer가 옵니다.</li>
  <li><strong>Request</strong>: PC가 어느 Offer를 받을지 명시해서 broadcast. 다른 서버는 자기 offer가 안 골라졌음을 알게 됩니다.</li>
  <li><strong>Ack</strong>: 선택된 서버가 lease 시간·gateway·DNS·옵션을 동봉해 확정. 이때부터 PC가 IP 사용 가능.</li>
</Steps>

## Lease 시간과 T1 · T2

받은 IP는 영구가 아닙니다. lease 시간 안에 *갱신*해야 합니다.

<KV>
  <KVItem term="T1 (50%)">lease의 절반 시점. PC가 *자기 서버에* unicast로 renewal 시도.</KVItem>
  <KVItem term="T2 (87.5%)">renewal 실패 시 *어느 서버든* broadcast로 rebind 시도.</KVItem>
  <KVItem term="Expire (100%)">아무도 응답 없으면 IP 반납. 처음부터 Discover 다시.</KVItem>
</KV>

가정 공유기는 보통 lease 24h, 기업 환경은 1~8h. **너무 길면**: 떠난 단말의 IP가 풀에 안 돌아와 IP 부족. **너무 짧으면**: 갱신 트래픽·서버 부하 증가.

## DHCP Snooping과 DAI

DHCP는 인증이 없습니다. 누구든 "내가 DHCP 서버야"라고 응답할 수 있습니다.

**DHCP Snooping** (스위치 기능):
- 스위치 포트를 *trusted* / *untrusted*로 구분
- untrusted 포트에서 오는 *서버측* 응답(Offer / Ack)은 차단
- trusted 포트(진짜 서버 연결)에서 오는 응답만 통과
- 스위치는 동시에 *IP↔MAC↔포트* binding table 작성

**Dynamic ARP Inspection (DAI)**:
- 같은 binding table을 ARP에도 적용
- 위조 ARP가 binding과 다르면 차단
- DHCP snooping + DAI = ARP 스푸핑 방어의 짝

<Callout>
  DHCP snooping을 켜고 trusted 포트 지정을 빼먹으면 <strong>진짜 서버 포함</strong> 모든 서버 응답이 차단됩니다. 신규 단말의 Discover가 시간 초과로 풀에 못 들어가는 사고. 항상 trusted 포트 먼저.
</Callout>

  <PC tone="bad" title="Rogue DHCP server">
